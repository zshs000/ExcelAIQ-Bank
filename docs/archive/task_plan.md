## 已知决策
- 本轮只做“分布式系统正确性/微服务边界”专项审查，不展开风格类、通用安全类和前端问题。
- 重点覆盖所有下游微服务调用点，以及 MQ、Redis、OSS、数据库状态机等跨边界交互。
- 文档只作为旁证；若文档与源码冲突，一律以当前源码实现为准。
- 以代码现状为准，现有文档只作为旁证，不直接采信。
- 优先关注 `eaqb-question-bank`、`eaqb-excel-parser`、`eaqb-auth`、`eaqb-user`，其余模块作为支撑服务理解。
- 最终要新增独立文档目录，集中输出审查报告、核心链路和问题清单。
- 面试优化优先补“可靠性、可解释性、可验证性”，不继续横向扩新业务。

## 风险与疑点
- 这次问题类型偏“系统性正确性”，很多结论需要串联 controller/service/feign/filter/consumer/mapper/test 才能成立，阅读范围会比较大。
- 历史文档可能描述过期设计，例如旧版 Redis 中转、旧版签名链路或旧版补偿方案，不能直接采信。
- 某些“下游错误被掩盖”的问题不一定表现为显式 bug，可能是异常包装、统一返回体或日志缺证据导致的可观测性问题。
- 工作区已经存在大量未提交改动，本轮只能在不破坏现有改动的前提下做审查与文档落盘。
- 项目内已有较多说明文档，可能存在“设计文档”和“实际代码”不一致的情况，需要以代码为准。
- 需要继续确认 AI 链路是否只是“消息契约预留 + mock 联调”，还是已经具备真正可用的生产闭环。
- 用户本轮明确要求“完整阅读整个项目”，因此不能只抽查关键类，需要覆盖源码、测试、配置与文档，但结论仍以代码为主。
- 用户明确暂不考虑“打印敏感信息、硬编码配置”两类问题；这两类只作为背景噪音，不进入本轮主要问题清单。
