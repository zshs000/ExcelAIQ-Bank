# 题目审核链路重构总结

## 1. 这次重构到底在收什么

这次不只是把 `GENERATE` 和 `VALIDATE` 两种模式拆开，而是顺手把整条异步链路的“身份、阶段、幂等”一起补齐。

旧链路最大的几个问题是：

1. 题目状态既想表达业务进度，又想表达 MQ 是否真的发出，职责混在一起。
2. AI 回包是否有效，依赖 `question.process_status == PROCESSING` 这种脆弱前提。
3. `VALIDATE` 虽然语义上已经和 `GENERATE` 区分开了，但审核记录还没有完全和本次处理任务绑定。
4. 回包重复消费没有一个稳定的 inbox 幂等锚点。

所以这轮真正的目标是：

**题目主表只表达业务态，异步真相交给 task/outbox/inbox。**

---

## 2. 最终落下来的结构

### 2.1 题目主表

`t_question_bank` 继续保留为业务聚合根，只关心用户视角下的状态：

- `WAITING`
- `DISPATCHING`
- `PROCESSING`
- `REVIEW_PENDING`
- `COMPLETED`
- `PROCESS_FAILED`

同时保留：

- `lastReviewMode`：最近一次进入待审核时，来自 `GENERATE` 还是 `VALIDATE`

### 2.2 异步过程表

新增三类过程数据：

1. `question_process_task`
   - 一次业务处理任务
   - 记录 `taskId / questionId / mode / attemptNo / taskStatus / callbackKey / failureReason`
2. `question_outbox_event`
   - 一次待派发事件
   - 记录 `taskId / eventStatus / dispatchRetryCount`
3. `question_callback_inbox`
   - 一次回包消费幂等记录
   - 记录 `callbackKey / taskId / consumeStatus`

这里最关键的点是：

- `attemptNo` 是业务重试次数
- `dispatchRetryCount` 是同一 outbox 事件的发送重试次数

这两个概念彻底分开了。

### 2.3 校验记录

`question_validation_record` 继续保留，并新增 `task_id`。

这意味着 `VALIDATE` 模式下的 AI 建议，不再只是“某题的一条建议”，而是明确属于“某次校验任务的建议”。

---

## 3. 现在主链路怎么跑

### 3.1 发送阶段

发送不再是“改状态后立刻赌 MQ”，而是先落本地事实，再派发：

1. 题目 `WAITING -> DISPATCHING`
2. 创建 `question_process_task`，状态 `PENDING_DISPATCH`
3. 创建 `question_outbox_event`，状态 `NEW`
4. dispatcher 发送 MQ
5. 发送成功后：
   - outbox: `NEW -> SENT`
   - task: `PENDING_DISPATCH -> DISPATCHED`
   - question: `DISPATCHING -> PROCESSING`

所以 `DISPATCHING` 的含义现在很清楚：

**本地已经承认“要派发这件事”，但还没确认消息已经可靠进入 AI 链路。**

### 3.2 回包阶段

AI 回包不再先看题目是不是 `PROCESSING`，而是先认任务身份：

1. 用 `callbackKey` 进 inbox 做幂等
2. 用 `taskId` 找到本次任务
3. 再决定是否落题目、任务、校验记录

这里的好处很直接：

- 即使题目还停在 `DISPATCHING`
- 只要 `taskId` 是当前有效任务
- 回包也能继续处理

也就是说，原来那个“消息刚发出成功，但本地还没来得及把题目改成 `PROCESSING`，结果回包先到了”的窗口，被 task 身份兜住了。

### 3.3 成功回包

`GENERATE`：

1. AI 答案写回题目
2. 题目进入 `REVIEW_PENDING`
3. `lastReviewMode = GENERATE`
4. task 标记 `SUCCEEDED`
5. inbox 标记 `PROCESSED`

`VALIDATE`：

1. 不覆盖原答案
2. 新增一条带 `taskId` 的校验记录
3. 题目进入 `REVIEW_PENDING`
4. `lastReviewMode = VALIDATE`
5. task 标记 `SUCCEEDED`
6. inbox 标记 `PROCESSED`

### 3.4 失败回包

1. 题目进入 `PROCESS_FAILED`
2. task 标记 `FAILED`
3. `failureReason` 落到 task
4. inbox 标记 `PROCESSED`

这意味着“为什么失败”终于不只在日志里，而是变成了过程数据。

---

## 4. 审核语义现在是什么

### 4.1 GENERATE

- `APPLY_AI`：采纳 AI 生成结果，题目 `COMPLETED`
- `REJECT`：驳回本次生成结果，清空生成答案，题目回到 `WAITING`

### 4.2 VALIDATE

- `KEEP_ORIGINAL`：保留原答案，题目 `COMPLETED`
- `APPLY_AI`：用 AI 建议覆盖原答案，题目 `COMPLETED`
- `REJECT`：驳回本次校验结果，题目回到 `WAITING`

这里还补了一个很重要的收口：

**人工审核时，优先按当前 pending 记录绑定的 `taskId` 去取校验记录，而不是只按 `questionId` 猜最新一条。**

这样做的意义是，后面哪怕一题多次校验保留历史，也不会把旧建议和新审核串起来。

---

## 5. 这轮重构真正解决了什么

### 5.1 解决了“回包先到”的状态窗口

以前是拿题目状态当异步真相，窗口天然存在。

现在是拿 `taskId` 当异步真相，题目状态只是业务外观。

### 5.2 解决了回包重复消费问题

现在 callback 会先过 inbox：

- 已处理过：直接忽略
- 首次处理：写入 `RECEIVED`
- 处理完成：推进为 `PROCESSED`

### 5.3 解决了 VALIDATE 结果污染主数据的问题

`VALIDATE` 回包只写校验记录，不直接改正式答案。

AI 建议和题目正式答案终于彻底分开了。

### 5.4 让失败原因可追踪

以前失败原因主要在日志里。

现在失败原因会落到 task，后续前端展示、排障、补偿都更好做。

---

## 6. 还没做完的部分

这轮先把主契约和主链路收口了，下面这些可以下一篇再展开：

1. outbox 定时扫描和补偿派发
2. inbox / task / outbox 的数据库唯一约束与索引
3. “同一题同一时刻只能有一个有效 task”的数据库级约束
4. 更多运维视角的监控和手工补偿入口

现在的状态可以概括成一句话：

**模式语义已经拆开，异步骨架也立住了；下一步再把补偿和运维层补满。**
