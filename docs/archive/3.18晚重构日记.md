# 3.18晚重构日记：从验证码并发复用到登录凭证解耦

## 起点：只是想修一个验证码并发问题

3 月 18 日晚上的重构，最开始其实非常具体，就是为了修短信验证码登录里的一个并发问题。

原始实现很朴素：

1. 根据手机号拼出 Redis Key
2. 从 Redis 取出验证码
3. 和用户提交的验证码比较
4. 一致则继续登录，并删除 Redis 里的验证码

单看流程没有问题，但只要把场景放到并发下，这套逻辑就不成立了。两个请求如果几乎同时到达，就可能同时读到同一个旧值。结果就是两个请求都认为自己校验成功，同一份验证码被重复使用。

这个问题后来越想越熟悉，因为它和传统秒杀场景的本质其实非常接近：都是在竞争一个只能成功消费一次的资源。

- 秒杀里，被消费的是库存
- 验证码登录里，被消费的是验证码的“成功校验资格”

换句话说，库存是“可售次数有限的资源”，验证码是“可成功消费一次的资源”。它们看起来业务完全不同，但底层矛盾是一样的，都是并发下的原子消费问题。

## 第一轮修复：用 Lua 做原子消费

既然问题本质是“比较”和“删除”分离导致的非原子性，那么最直接的修复方式，就是把这两个动作合成一个不可分割的操作。

这次重构里，最终采用的是 Redis Lua 脚本：

- 先在 Redis 内部读取当前验证码
- 判断是否等于用户提交的验证码
- 如果一致，立即删除
- 如果不一致，直接返回失败

这样一来，验证码就不再是“先查出来，再决定删不删”，而是变成了“在 Redis 内部完成一次原子消费”。

这个修改很关键，因为它真正修复的是语义，而不是代码表面：

- 验证码不是普通字符串
- 它本质上是一种一次性资源
- 一次性资源就必须按一次性资源的方式消费

做到这一步时，其实问题已经在并发层面被修好了。

## 审查过程中，问题又往前走了一步

本来按正常节奏，做到 Lua 这里完全可以收工。

但在继续审查这条链路的时候，脑子里突然蹦出来一句以前看《凤凰架构》时印象特别深的话：**一切服务都是不可靠的。**

这句话一出来，视角就变了。

之前只盯着验证码并发复用，关注点是：

- 同一份验证码不能被两个请求同时用掉

但如果把链路看完整，会发现还有另一层问题：

1. Lua 已经把验证码成功消费掉了
2. 后续还要调用下游服务，比如注册用户、建立登录态
3. 可这些下游服务本身并不可靠，可能超时、失败、返回异常

那问题就来了：

**如果验证码已经删了，但下游服务失败了，怎么办？**

这时候并发问题虽然解决了，但用户体验和链路可靠性的问题又冒出来了。用户明明验证码输对了，却因为后面某个服务失败，只能重新获取一条新的验证码。

## 中间方案：先锁住验证码，登录成功后再删

顺着这个问题继续想，一个很自然的中间方案就出现了：

既然“先删”可能导致后面失败无法恢复，那是不是可以先不删，而是先把验证码锁住，等整个登录流程真的走完之后再删除？

这个思路初看很合理，因为它试图同时兼顾两件事：

- 保证验证码不会被并发重复使用
- 又避免验证码过早删除，导致下游失败后用户必须重新获取

但这个方案越想越发现工程代价很高，因为它会把验证码的状态机复杂化。

原来验证码只有两种状态：

- 存在
- 不存在

一旦引入“锁住验证码”，状态就开始膨胀：

- 可用
- 已锁定
- 已消费
- 锁超时待释放
- 请求中断后的恢复状态

随之而来的工程问题也会非常多：

- 锁由谁释放
- 锁多久超时
- 下游失败后怎么恢复
- 客户端重试时怎么处理
- 服务实例异常退出后，锁状态是否会卡死

也就是说，这个方案虽然比“直接删除”更照顾体验，但它把很多流程状态都压到了验证码本身上。

而验证码本来不应该承担这么多职责。

## 最终方向：解耦验证码与后续登录流程，引入凭证

继续往下推，最后发现更干净的思路不是“如何更聪明地保留验证码”，而是**不要让验证码承担后续登录流程的状态。**

更合理的做法是把这条链路拆成两段：

1. 验证码只负责证明“这次校验通过”
2. 一旦校验成功，就签发一个短期凭证，由这个凭证去承接后续注册、登录、发 token 等流程

这样验证码和后续流程就真正解耦了。

验证码的语义重新变得非常纯粹：

- 它只负责一次性验证
- 验证成功后，它的使命就结束

后面的复杂性则交给短期凭证来处理：

- 下游服务失败，可以围绕凭证做重试或补偿
- 登录链路可以拆阶段推进
- 验证码不必再承载锁、恢复、超时释放这些额外状态

换句话说，最终不是在“删验证码”这件事上继续打补丁，而是重新拆分职责：

- 验证码负责认证
- 凭证负责承接流程
- 最终登录态负责业务完成

## 这次重构最大的收获

这次重构最开始只是修一个小并发点，但最后沉淀下来的不是一个 Lua 脚本，而是一条很完整的思考链路：

1. 从业务 Bug 出发，发现验证码存在并发复用问题
2. 抽象出它和秒杀防超卖的共同本质，都是一次性资源的原子消费
3. 用 Lua 在 Redis 内部完成原子校验加删除
4. 进一步意识到，即使原子消费成功，下游服务仍然可能不可靠
5. 思考过“先锁后删”的过渡方案，但发现会引入更多工程复杂度
6. 最终得出更清晰的方向：通过短期凭证解耦验证码校验与后续登录流程

如果用一句话总结这次重构，那就是：

**问题表面上是验证码并发复用，往下深挖后，本质上其实是“一次性资源的原子消费”与“不可靠服务链路”之间的协调问题。**

而这也是这次 3 月 18 日晚重构最有价值的地方。它不是停在“把 bug 修掉”，而是顺着问题一路把系统边界重新想清楚了。
