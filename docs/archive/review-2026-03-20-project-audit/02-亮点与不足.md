# 亮点与不足

## 1. 亮点

### 1.1 Excel 链路的分层比表面看起来成熟

- 上传和解析分成两段，不把“校验、存文件、解析、入库”塞到一个接口里。
- 模板校验失败会持久化错误明细，而不是直接返回一次性报错，这对前端体验是加分项。
- `parse-by-id` 已经落地条件更新抢占，重复解析问题被真正收口，不再只是文档说明。

### 1.2 题目状态机已经从“口头约定”变成代码事实

- `WAITING -> PROCESSING -> REVIEW_PENDING -> COMPLETED` 主流程清晰。
- AI 失败、人工驳回、重试等支路也都有显式状态。
- 发送、回写、审核都改成基于预期状态的更新，避免旧状态覆盖新状态。

### 1.3 AI 契约意识比一般 demo 项目强

- `QuestionMessage` 和 `AIProcessResultMessage` 明确建模了请求/回包字段。
- 同时兼容旧字段和新字段，说明作者已经做过一次链路收敛，不是纯一次性实验代码。

### 1.4 测试覆盖在重点服务上有投入

- `ExcelFileServiceImplTest` 不只是 happy path，还覆盖了越权、文件不存在、导入失败、重复解析抢占等场景。
- `QuestionServiceImplTest` 对发送、删除、编辑、审核等关键行为都有单测。
- `UserServiceImplTest` 至少把“已存在用户”“并发注册兜底”“默认密码加密落库”这些关键行为钉住了。

## 2. 不足

### 2.1 账户体系安全策略偏弱

- 自动注册用户使用固定默认密码。
- 密码登录没有“密码未初始化”或“首次登录必须改密”的保护。
- 这会把验证码登录带来的便利性，反过来变成口令安全隐患。

### 2.2 鉴权边界完全押注网关

- 业务服务自身没有再校验 token，只是信任请求头 `userId`。
- 这种方案在“所有业务服务都只允许内网访问”时勉强成立，但对真实部署要求很高。

### 2.3 配置治理明显落后于代码治理

- 仓库中有 Redis、Nacos、MinIO、MySQL、Druid 管理台等凭据。
- 这类问题不会在单测里暴露，但上线前必须处理，否则一旦仓库外流，影响范围很大。

### 2.4 项目目标和仓库内容仍有落差

- README 仍强调“Python AI Service + Redis 缓存 AI 结果”，但当前代码已经改成消费者直接回写数据库。
- 本仓库里没有真正的 AI 服务实现，因此更像“可扩展 AI 异步接口”，而不是“完整 AI 题目系统”。

### 2.5 微服务拆分对当前业务略重

- 业务本体并不复杂，核心其实就是文件导入和题目处理。
- 但围绕它拆了网关、认证、用户、Excel、题库、OSS、ID 服务。
- 对学习 Spring Cloud Alibaba 是加分项；对纯业务交付来说，系统复杂度偏高。

## 3. 综合评价

如果把它当成“微服务工程练习项目”，这套设计是有说服力的，尤其是链路收口、状态机和测试比很多练手仓库扎实。

如果把它当成“即将对外提供真实服务的题库平台”，当前最需要补的不是再加新功能，而是：

1. 账户与密码策略
2. 服务间信任边界
3. 仓库凭据治理
4. 文档收敛
