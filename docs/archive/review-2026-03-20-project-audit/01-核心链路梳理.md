# 核心链路梳理

## 1. 模块地图

- `eaqb-gateway`
  - 网关路由、Sa-Token 登录校验、把 `userId` 注入下游请求头
- `eaqb-auth`
  - 验证码发送、验证码/密码两种登录策略、登录态建立、改密入口
- `eaqb-user`
  - 用户注册、按手机号查用户、改密、角色缓存
- `eaqb-excel-parser`
  - Excel 上传校验、预上传错误记录、按 `fileId` 解析导入
- `eaqb-question-bank`
  - 题目 CRUD、批量导入、发送 AI、AI 回包回写、人工审核
- `eaqb-oss`
  - 文件上传和预签名访问链接
- `eaqb-distributed-id-generator`
  - 用户、文件、题目等业务 ID 分配

## 2. 登录与鉴权链路

### 2.1 主流程

1. 前端访问网关。
2. 网关 `SaTokenConfigure` 对除登录/发验证码外的路径执行登录校验。
3. 登录请求进入 `auth`。
4. `auth` 根据登录类型分派到验证码登录或密码登录策略。
5. 登录成功后由 `StpUtil.login(userId)` 建立登录态。
6. 之后的业务请求再经过网关，由 `TrustedInternalCallGatewayFilter` 重写内部请求头并追加受信签名，已登录时同时透传真实 `userId`。
7. 下游服务通过 `TrustedInternalRequestFilter` 先验签，再把受信请求头里的 `userId` 放入 `ThreadLocal`，业务层从 `LoginUserContextHolder` 读取。

## 3. Excel 上传、解析、导入链路

### 3.1 上传阶段

1. `excel-parser/upload` 接收文件。
2. `ExcelFileServiceImpl.uploadAExcel()` 依次做：
   - 文件大小校验
   - 扩展名校验
   - 魔术头校验
   - EasyExcel 模板校验
3. 模板失败时：
   - 生成 `preUploadId`
   - 写 `t_excel_pre_upload_record`
   - 返回失败摘要，前端可再查明细
4. 模板成功时：
   - 调 `oss` 上传文件
   - 调 `id-generator` 生成 `fileId`
   - 写 `t_file_info`
   - 文件状态记为 `UPLOADED`

### 3.2 解析导入阶段

1. 前端拿 `fileId` 调 `parse-by-id`。
2. `ExcelFileServiceImpl.parseExcelFileById()`：
   - 加载文件记录并校验归属
   - 通过 `tryMarkParsing(fileId, userId)` 抢占解析资格
   - 调 `oss` 获取下载地址
   - 下载 Excel
   - 解析成题目 DTO 列表
   - 转成 `question-bank` 批量导入请求
3. 导入结果决定文件状态：
   - 成功 `PARSED`
   - 失败 `FAILED`

### 3.3 当前状态与语义

- 文件状态
  - `UPLOADED`
  - `PARSING`
  - `PARSED`
  - `FAILED`
- 返回语义
  - 文件不存在、无权限、下载失败：外层失败
  - 题库导入失败：外层接口成功，但 `ExcelProcessVO.processStatus = FAILED`

## 4. 题目发送 AI、回写、审核链路

### 4.1 题目导入后的初始态

- `question-bank` 批量导入会给每道题分配唯一 ID，并统一写成 `WAITING`
- 关键代码
  - `eaqb-question-bank/eaqb-question-bank-biz/src/main/java/com/zhoushuo/eaqb/question/bank/biz/service/impl/QuestionServiceImpl.java:246-317`

### 4.2 发送到 AI

1. 用户访问 `question-bank/api/question/send-to-queue`
2. `QuestionServiceImpl.sendQuestionsToQueue()`：
   - 校验当前用户
   - 校验模式 `GENERATE / VALIDATE`
   - 过滤不属于当前用户、状态不合法、模式不匹配的题目
   - 用条件更新把题目从 `WAITING` 推到 `PROCESSING`
   - 发 RocketMQ 消息
3. 如果 MQ 不可用但 `mock-enabled=true`：
   - 本地直接模拟 `AI_SUCCESS`
   - 题目推进到 `REVIEW_PENDING`
4. 如果消息发送失败：
   - 回滚 `PROCESSING -> WAITING`

### 4.3 AI 回包

1. `AIProcessResultConsumer` 监听 `AIProcessResultTopic`
2. 成功回包：
   - `PROCESSING -> REVIEW_PENDING`
   - `GENERATE` 默认覆盖答案
   - `VALIDATE` 默认不覆盖原答案
3. 失败回包：
   - `PROCESSING -> PROCESS_FAILED`

### 4.4 人工审核

1. `reviewQuestion()` 支持 `APPROVE / REJECT`
2. 合法状态流转：
   - `REVIEW_PENDING -> COMPLETED`
   - `REVIEW_PENDING -> WAITING`

## 5. 现在这套链路的真实定位

- 不是“完整 AI 产品”
- 更准确的说法是：
  - Java 微服务主流程已打通
  - AI 通过消息契约异步接入
  - 本仓库内保留了本地 mock 和回包兼容逻辑
  - 真正的外部 AI 服务仍在仓库之外
