# Excel AI Q-Bank 项目总览审查

## 1. 审查范围

- 仓库整体结构、模块职责与服务边界
- 核心业务链路
  - 登录与鉴权
  - Excel 上传、解析、导入
  - 题目发送 AI、回写、人工审核
- 重点模块
  - `eaqb-excel-parser`
  - `eaqb-question-bank`
  - `eaqb-auth`
  - `eaqb-user`
  - `eaqb-gateway`
- 配置、测试、现有文档与代码一致性

## 2. 一句话结论

这是一个“用题库业务承载微服务工程能力练习”的项目，主链路已经比旧文档描述更稳，尤其是 Excel 导入和题目状态机的收口做得不错；但安全边界、密钥治理和账户策略仍有明显短板，距离“可放心对外暴露的生产系统”还有一段距离。

## 3. 项目亮点

- Excel 导入被拆成“上传校验”和“按 `fileId` 解析导入”两段，失败信息可以按 `preUploadId` 回查，交互语义清楚。
- `excel-parser` 已对重复解析做条件更新抢占，避免同文件被并发重复导入。
- `question-bank` 已把发送 AI、AI 回写、人工审核、编辑、删除统一到状态机 + CAS 条件更新语义里，主流程一致性较好。
- `question-bank`、`excel-parser`、`user` 的核心服务都有像样的单测，`excel-parser` 还用本地 HTTP Server 模拟 OSS 下载，测试思路务实。
- AI 消息模型和回包模型已经明确区分 `GENERATE / VALIDATE` 两种模式，并保留旧字段兼容逻辑。

## 4. 主要不足

- 新用户固定默认密码 `123456`，同时系统允许手机号 + 密码直接登录，默认口令风险较高。
- 下游服务普遍直接信任来自请求头的 `userId`，一旦服务端口被旁路访问，鉴权边界会失效。
- 多个配置文件和代码生成配置里存在明文/可逆凭据，仓库密钥治理较弱。
- 验证码、网关、上下文过滤器保留了较多敏感或调试日志，运行期暴露面偏大。
- AI 模块目前更像“契约 + MQ 接口 + 本地 mock 预留”，仓库内并没有真正的 AI 服务实现。
- 根 `README.md` 已明显落后于代码现状，不能直接当作当前架构说明。

## 5. 审查结论分级

### 高优先级

1. 新用户默认弱口令
2. 网关外旁路访问导致的 `userId` 伪造风险
3. 仓库凭据暴露

### 中优先级

1. 验证码明文日志
2. 过量调试日志
3. AI 能力仍偏“预留式集成”，生产闭环不足
4. 文档与代码存在漂移

### 低优先级

1. 认证与配置边界的自动化测试还不够
2. 部分模块命名、Topic 命名仍偏 demo 化，例如 `TestTopic`
