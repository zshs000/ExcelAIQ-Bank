# Excel 导入 contentHash 风险说明

## 1. 背景

当前导入链路采用分块幂等：

- excel-parser 计算 `contentHash` 并上送。
- question-bank 通过 `(batchId, chunkNo, rowCount, contentHash)` 判断分块是否重复。

当前 `contentHash` 的输入串行化方式是“字段值 + `\n` 分隔”。

## 2. 风险机制

风险不在 SHA-256 算法本身，而在哈希前的输入串行化存在边界歧义。

当字段内容本身包含换行时，不同字段边界可能被拼成相同字节流，进而导致：

- 不同业务内容得到相同 `contentHash`。
- 下游按现有判重条件误判为 `duplicate`。

## 3. 可能后果

出现“不同内容同 hash”时，可能发生：

1. 第二次请求被当成 duplicate 成功返回。
2. 不触发冲突错误，也不标记批次失败。
3. 第二次内容不落库，最终保留旧内容。
4. 链路表面成功，但数据与预期不一致（静默数据错误）。

## 4. 高风险触发条件

以下情况会放大风险：

- 字段中允许出现换行或分隔符字符。
- 重试时依赖 `rowCount + contentHash` 做唯一判定。
- 下游不重算哈希、仅信任上游上传值。

## 5. 建议修复方向

1. 将哈希输入编码改为无歧义格式（例如长度前缀 + 字段字节）。
2. 增加 `hashVersion`，避免规则演进时混判。
3. 下游按同规则重算并校验，逐步从“单边信任”升级为“一致性校验”。
