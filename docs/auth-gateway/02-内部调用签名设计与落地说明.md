# 内部调用签名设计与落地说明

## 1. 这份文档解决什么问题

这份文档记录本项目在“防止用户绕过 gateway 直接访问下游服务”这个问题上的完整思考和实际落地。

目标不是只解释“代码怎么写”，而是把下面几件事讲清楚：

- 一开始为什么只靠部署层隔离
- 为什么后来发现仅靠部署层不够
- 为什么最终选择用全局过滤器 / 全局拦截器统一落地
- 当前内部调用签名方案到底做到了什么
- 当前还没有做到什么，为什么暂时没有继续做
- 这套思路在面试里应该怎么表达

## 2. 初版方案：主要依赖部署层隔离

项目初版的隐含前提是：

- 对外只暴露 gateway
- 下游微服务只允许内网访问
- gateway 完成登录校验后，把 `userId` 写入请求头
- 下游服务只要看到 `userId`，就把它写入上下文

这个方案不是完全错误。

在很多中小型微服务项目里，这其实是一个很常见的起步方案。因为部署层如果做得严格，下游服务确实不应该被外部直接访问。

但是这个方案有一个明显前提：

> 默认部署一定正确。

而工程上不能完全信任这个前提。

只要出现下面任意一种情况，这个前提就会破：

- 运维把下游服务端口暴露错了
- 安全组或网关规则配置错误
- 某个服务被旁路访问
- 测试环境、灰度环境、临时联调环境暴露不规范

一旦前提被破，而下游服务又裸信任 `userId` 请求头，就会出现典型风险：

- 外部调用方自己伪造 `userId`
- 下游服务把它当成真实登录用户
- 直接导致越权

所以，部署层隔离仍然需要保留，但不能作为唯一防线。

## 3. 改进目标：在代码层补一层“内部调用验真”

后续改造的核心目标是：

> 下游服务不再裸信任请求头，而是只信任“验签通过的内部请求”。

也就是说：

- 不是说外部请求永远进不来
- 而是即使它进来了，只要不是受信内部服务发出来的，也不能被当成合法内部调用

这就把“信任基础”从“默认部署没问题”，升级成了：

- 部署层尽量保证只暴露 gateway
- 代码层再校验这次请求是不是受信内部服务发出来的

这是一种更稳的双层防护思路。

## 4. 为什么适合用全局过滤器 / 全局拦截器落地

这个问题本质上不是某一个 controller、某一个 service 的业务逻辑问题，而是一个典型的横切关注点。

它和下面这些能力很像：

- 全局异常处理
- 全局日志
- 统一响应
- 参数校验
- 上下文透传
- 限流
- 幂等

这类能力最适合做法不是“每个接口都手写一遍”，而是统一收口。

本项目最终选择放在三处横切组件中统一处理：

1. gateway 出站全局过滤器
2. 下游服务入站全局过滤器
3. Feign 出站全局拦截器

这样做的优点是：

- 不需要每个业务接口单独改造
- 不容易漏某个服务
- 逻辑集中，后续增强成本低
- 更符合“横切能力下沉”的工程思路

这也是这次改造里一个非常值得讲的点：

> 全局过滤器 / 全局拦截器的价值，不只是做日志和鉴权，也很适合做服务间信任边界治理。

## 5. 本项目当前的落地组件

当前代码里，内部调用签名链路主要通过这几处落地：

- gateway 出站签名：
  - `eaqb-gateway/src/main/java/com/zhoushuo/eaqb/gateway/filter/TrustedInternalCallGatewayFilter.java`
- 下游入站验签：
  - `eaqb-framework/eaqb-spring-boot-starter-biz-context/src/main/java/com/zhoushuo/framework/biz/context/filter/TrustedInternalRequestFilter.java`
- Feign 出站签名：
  - `eaqb-framework/eaqb-spring-boot-starter-biz-context/src/main/java/com/zhoushuo/framework/biz/context/interceptor/TrustedFeignRequestInterceptor.java`
- 公共签名工具：
  - `eaqb-framework/eaqb-common/src/main/java/com/zhoushuo/framework/commono/util/InternalRequestAuthUtil.java`
- 公共请求头常量：
  - `eaqb-framework/eaqb-common/src/main/java/com/zhoushuo/framework/commono/constant/GlobalConstants.java`

从职责上看：

- `TrustedInternalCallGatewayFilter`
  - 负责 gateway 转发时重写内部调用头并生成签名
- `TrustedFeignRequestInterceptor`
  - 负责微服务之间 Feign 调用时生成签名
- `TrustedInternalRequestFilter`
  - 负责下游收到请求后先验签，再决定是否信任 `userId`

## 6. 当前签名方案是什么

本项目当前实现的是：

> 基于共享密钥的 HMAC-SHA256 内部请求签名

它不是 JWT，也不是 OAuth。

它更像一种：

> 服务间请求防伪签名

### 6.1 什么叫“共享密钥”

所谓共享密钥，意思不是公开共享，而是：

- gateway 知道这个密钥
- 其他受信内部服务也知道这个密钥
- 外部调用方不知道这个密钥

在代码里，这个密钥通过配置项注入：

- `internal.auth.secret`

### 6.2 什么叫 HMAC-SHA256

可以先把它简单理解成：

> 用一个只有内部服务知道的密钥，对固定请求内容做一次“带密钥的哈希签名”。

如果只是普通哈希，谁都可以算。

而 HMAC 的关键点在于：

- 只有知道密钥的人，才能算出同样的签名

本项目当前使用：

- HMAC
- 底层哈希算法为 `SHA-256`

所以整体叫：

- `HMAC-SHA256`

### 6.3 当前参与签名的字段

当前版本签进去的字段有：

- `callerService`：调用方服务标识
- `userId`：当前用户 ID，没有则为空
- `timestamp`：请求时间戳（毫秒）
- `method`：HTTP Method，例如 `GET`、`POST`

调用方会把这些字段按固定顺序拼成一个 payload，然后用共享密钥做 HMAC-SHA256。

### 6.4 签名放在哪里

最终生成的签名以及辅助字段，会放到请求头中：

- `X-Caller-Service`
- `X-Call-Timestamp`
- `X-Call-Signature`
- `userId`

其中：

- `X-Caller-Service`
  - 表示“是谁在调我”
- `X-Call-Timestamp`
  - 用于时间窗口校验
- `X-Call-Signature`
  - 用于验签
- `userId`
  - 只有在有登录上下文时才透传

同时，gateway 或 Feign 出站时会先清掉这些同名头，再由系统自己写入，避免外部自带伪造值。

## 7. 签名和验签完整流程

### 7.1 gateway -> 下游

1. gateway 收到外部请求
2. 完成登录态获取
3. 读取自己的服务名 `spring.application.name`
4. 取出当前用户 ID、当前时间戳、HTTP method
5. 按固定顺序拼接 payload
6. 用共享密钥做 `HMAC-SHA256`
7. 把签名和辅助字段写到请求头
8. 转发给下游服务

### 7.2 微服务 A -> 微服务 B

1. 服务 A 先通过入站过滤器验签上游请求
2. 如果通过，再从上下文中拿到 `userId`
3. 当服务 A 用 Feign 调服务 B 时
4. Feign 出站拦截器读取：
   - A 自己的服务名
   - 当前 `userId`
   - 当前时间戳
   - 本次 Feign 请求的 HTTP method
5. 再次生成新的签名
6. 把新签名放到请求头
7. 服务 B 入站后再重新验签

注意：

> 签名是逐跳重新生成、逐跳重新验证的，不是整条链路只签一次。

## 8. 下游服务如何验证

下游的统一入站过滤器收到请求后，会先读取请求头里的：

- `X-Caller-Service`
- `X-Call-Timestamp`
- `X-Call-Signature`
- `userId`

然后做两步校验：

### 8.1 时间窗口校验

先看请求时间戳是否在允许范围内。

本项目当前默认窗口是：

- `300` 秒，也就是 5 分钟

配置项为：

- `internal.auth.max-skew-seconds`

### 8.2 重算签名

如果时间窗口合法，下游会：

1. 按和上游完全相同的规则重新拼接 payload
2. 用同一个共享密钥重新做 `HMAC-SHA256`
3. 比较重算结果和 `X-Call-Signature` 是否一致

如果一致，说明：

- 这些字段大概率来自持有共享密钥的内部服务
- 中途没有被篡改

这时下游才会信任 `userId`，并把它写入上下文。

如果不一致，或者时间戳已经过期，就直接拒绝请求。

## 9. 为什么签 method，而没有先签 path / query / body

这是一个非常关键的设计取舍点。

理论上更严格的签名体系，往往还会把这些内容也签进去：

- 请求 path
- query 参数
- body 摘要

这样能进一步防止请求参数被篡改。

但是本项目当前没有这么做，原因是：

### 9.1 method 比较稳定

同一跳请求里：

- gateway 出站看到的是 `POST`
- 下游入站看到的通常也是 `POST`

所以 `method` 作为签名字段比较稳定。

### 9.2 path 在 gateway 转发场景下不一定稳定

gateway 转发时可能存在：

- 路由前缀
- StripPrefix
- 路径重写

这样：

- gateway 签名时看到的 path
- 下游实际接收到的 path

不一定完全一致。

如果贸然把 path 签进去，很容易出现：

- 合法请求也验签失败

### 9.3 body 的实现复杂度更高

如果要把 body 也签进去，通常需要：

- 在 gateway 层可重复读取请求体
- 统一 body 序列化和规范化规则
- 避免因为空格、字段顺序、编码差异造成误判

对于当前项目来说，先把“防伪造请求头”这个核心问题收住更重要。

所以当前选择的是：

> 先落一个最小可用且稳定的版本，而不是一开始就做成完整开放平台级别的请求签名体系。

## 10. 时间窗口为什么默认是 5 分钟

当前默认值：

- `300` 秒

它不是安全最优值，而是一个偏保守的工程默认值。

这样选的原因主要有三点：

### 10.1 容忍时钟漂移

分布式环境下，不同机器之间可能存在轻微时钟偏差。

如果窗口太小，比如几秒钟，就容易误伤合法请求。

### 10.2 容忍网络抖动和重试延迟

真实请求链路里，可能存在：

- 网络抖动
- 调用重试
- 网关转发延迟

窗口适当放宽，更利于稳定落地。

### 10.3 安全和可用性的折中

如果窗口太大：

- 重放攻击的有效时间也会变长

如果窗口太小：

- 正常请求可能被误判为过期

所以当前项目先取了一个：

> 保守默认值

后续如果服务时间同步更稳定、链路更短、重试策略更可控，就可以继续收紧，比如压到 1 分钟甚至更短。

## 11. 当前版本为什么还不算完整防重放

当前版本已经做到了：

- 防止外部直接伪造 `userId` 请求头
- 防止很久以前抓包的旧请求再次使用

但还没有做到完整防重放。

### 11.1 什么是重放攻击

重放攻击不是“自己伪造一份新请求”，而是：

> 攻击者拿到一份真实合法请求后，原样再发一遍。

如果请求仍在时间窗口内，而系统又没有额外机制限制“这份请求只能用一次”，那么它有可能再次通过。

### 11.2 为什么当前版本挡不住“窗口内重放”

因为当前版本只有：

- 时间戳窗口

它能拦住：

- 过期请求

但拦不住：

- 在 5 分钟有效窗口内，把同一份请求重复发送多次

### 11.3 完整防重放通常怎么做

通常会额外加入：

- `nonce`

所谓 `nonce`，就是：

> 一次性随机值

做法通常是：

1. 请求里带一个唯一 `nonce`
2. `nonce` 也参与签名
3. 下游验签成功后，把 `nonce` 写入 Redis，并设置短 TTL
4. 如果同一个 `nonce` 再次出现，就拒绝

本质上，这和验证码消费很像：

- 验证码只能成功消费一次
- `nonce` 也应该只能成功消费一次

所以它们背后的抽象非常接近：

> 都是在处理“一次性资源”的消费问题。

### 11.4 为什么当前项目暂时没有继续做 nonce 防重放

不是不会做，而是做了取舍。

当前项目先优先解决的是：

- 绕过 gateway 伪造请求头的问题

这一步已经通过签名落地解决。

而 `nonce + Redis` 防重放虽然更完整，但会进一步引入：

- Redis 状态存储
- 原子去重逻辑
- 一次性消费语义
- 更多工程复杂度

结合当前系统规模和场景，这一步暂时没有继续堆上去。

更准确的说法是：

> 当前版本已经实现最小可用闭环；更强的防重放能力已评估，但暂未继续上复杂度。

## 12. 当前方案的优点和边界

### 12.1 优点

- 已经从“只信部署”升级成了“部署层 + 代码层双保险”
- 能显著降低绕过 gateway 伪造 `userId` 的风险
- 通过全局过滤器 / 全局拦截器统一落地，不容易漏
- 改造点集中，后续增强成本低
- 不需要每个 controller 单独写验签逻辑

### 12.2 当前边界

- 还没有 `nonce + Redis` 的完整防重放
- 还没有把 path / query / body 纳入签名
- 当前使用的是静态共享密钥
- 当前没有更细粒度的调用方白名单

所以，这套方案应该被描述成：

> 一个已经落地、能解决核心风险的最小可用版本。

而不是：

> 一个已经做到最高安全等级的终态方案。

## 13. 如果共享密钥泄露了怎么办

这是一个必须有的风险意识点。

如果共享密钥泄露，攻击者就能自己生成合法签名，这套签名机制的价值会大幅下降。

常见应对思路包括：

- 立即轮换密钥
- 支持双密钥过渡
- 缩小密钥影响面
- 增加调用方白名单
- 做异常审计和监控

### 13.1 双密钥过渡

为了避免滚动发布过程中服务互相打挂，通常不会简单粗暴“一刀切”。

更稳妥的方式是：

- 出站统一用新密钥签名
- 入站在过渡期同时兼容“新密钥”和“旧密钥”验签
- 所有服务升级完成后，再下线旧密钥

这就是典型的平滑密钥轮换思路。

## 14. 更强方案可以做到哪里

当前项目不需要一口气把所有高级方案都堆上去，但认知上可以知道后续方向。

可以继续增强的方向包括：

- `nonce + Redis` 防重放
- path / query / body 参与签名
- 调用方服务白名单
- 双密钥平滑轮换
- 按域拆分内部密钥，避免“一把钥匙开所有门”

再往上，如果系统规模更大、安全等级更高，还可以考虑：

- mTLS
- 服务网格

这里只提概念，不在当前项目里展开实现。

### 14.1 mTLS 是什么

可以把它理解成：

> 服务和服务之间通过双向证书认证彼此身份。

它比“共享密钥 + header 验签”更强，因为不再主要依赖应用层签名，而是把身份认证下沉到传输层。

### 14.2 服务网格是什么

可以把它理解成：

> 把服务间认证、流量治理、限流、重试、监控等横切能力，下沉到基础设施层统一处理。

这类方案通常更适合规模更大的微服务集群。

## 15. 为什么这套思路很适合面试讲

因为它不是只会背“对外只暴露 gateway”。

它体现了三层能力：

### 15.1 能看到问题

能意识到：

- 只靠部署层隔离不够
- 下游不能裸信任请求头

### 15.2 能落地实现

不是停留在口头方案，而是实际改成了：

- gateway 出站签名
- Feign 出站签名
- 下游入站验签

### 15.3 能做工程取舍

知道：

- 当前先实现最小可用闭环
- 防重放、双密钥、mTLS、服务网格等增强方向都存在
- 但不应该在当前项目里不加判断地全部堆上

这会让你的表达从“背八股”升级成：

> 我先解决核心问题，再评估增强路线。

## 16. 一段适合面试的表达模板

可以按下面这个结构讲：

“我一开始的方案主要依赖部署层隔离，对外只暴露 gateway，下游服务默认不对外开放。  
但我后来继续审视后发现，部署层并不能绝对信任，一旦端口误暴露或发生旁路访问，下游如果裸信任 `userId` 请求头就会有越权风险。  
所以我又在代码层补了一层内部调用签名。具体做法是：gateway 和服务间 Feign 调用统一使用共享密钥，对调用方服务标识、用户 ID、时间戳和 HTTP method 做 HMAC-SHA256 签名，把签名和辅助字段写到请求头；下游服务统一在入站过滤器里验签，只有验签通过才信任 `userId`。  
这样就算有人绕过 gateway 直接访问下游服务，只伪造 `userId` 也没用。  
当前版本已经解决了伪造请求头这个核心问题。更完整的防重放可以再加 nonce 和 Redis 一次性消费语义，但结合项目当前规模，我先落了最小可用闭环，没有继续堆复杂度。后续如果安全等级更高，还可以继续演进到双密钥轮换、调用方白名单，甚至 mTLS 或服务网格。” 

## 17. 最后一条结论

这次改造最重要的收获，不只是“加了签名”。

更重要的是形成了一个工程认知：

> 很多安全、日志、上下文、幂等等能力，都不应该散落在业务接口里，而应该优先考虑通过全局过滤器、全局拦截器、AOP、统一组件等方式集中治理。

这类“横切能力下沉”的思路，本身就是项目里非常值得总结和面试表达的工程亮点。
