# AI生成与校验契约 v1（任务化版本）

## 1. 目标

统一 Question-Bank 与 AI 的请求/回包契约，保证：

1. 同一主题可承载 `GENERATE` 和 `VALIDATE` 两类任务。
2. 每次异步处理都有明确任务身份，而不是只靠题目状态猜。
3. 回包可做稳定幂等，不怕重复消费。

---

## 2. 模式定义

1. `GENERATE`
   - 题目没有答案
   - AI 负责生成答案
2. `VALIDATE`
   - 题目已有答案
   - AI 负责判断答案质量，并返回结论与建议答案

---

## 3. 发送消息契约（Question-Bank -> AI）

Topic：`TestTopic`

```json
{
  "task_id": "80010001",
  "attempt_no": 1,
  "question_id": "123456789",
  "question_text": "什么是 JVM？",
  "mode": "VALIDATE",
  "current_answer": "Java 虚拟机"
}
```

字段说明：

1. `task_id`
   - 一次业务处理任务的唯一身份
   - AI 回包时必须原样带回
2. `attempt_no`
   - 第几次业务处理
   - 只在业务重试时增加
   - 不等于 MQ 派发重试次数
3. `question_id`
   - 题目 ID
4. `question_text`
   - 题目内容
5. `mode`
   - `GENERATE | VALIDATE`
6. `current_answer`
   - 仅 `VALIDATE` 有值
   - `GENERATE` 为空

---

## 4. 回包消息契约（AI -> Question-Bank）

Topic：`AIProcessResultTopic`

```json
{
  "task_id": "80010001",
  "attempt_no": 1,
  "callback_key": "80010001",
  "question_id": "123456789",
  "mode": "VALIDATE",
  "success_flag": 1,
  "ai_answer": "Java Virtual Machine",
  "validation_result": "FAIL",
  "reason": "术语不完整，建议补全英文全称"
}
```

字段说明：

1. `task_id`
   - 本次回包对应哪个任务
   - 是回包合法性的第一锚点
2. `attempt_no`
   - 本次任务的业务尝试次数
3. `callback_key`
   - 回包幂等键
   - 需要端到端稳定原样回传
   - 当前兼容逻辑下，若未显式传回，会退化为 `task_id`
4. `question_id`
   - 题目 ID
5. `mode`
   - `GENERATE | VALIDATE`
6. `success_flag`
   - `1/0`
7. `ai_answer`
   - `GENERATE` 下是生成答案
   - `VALIDATE` 下是建议答案
8. `validation_result`
   - `PASS | FAIL | UNCERTAIN | NA`
9. `reason`
   - 失败原因或补充说明

兼容说明：

1. 为兼容旧链路，后端仍可读取旧字段 `answer`、`error_message`。
2. 读取优先级：
   - 答案：`ai_answer` > `answer`
   - 原因：`reason` > `error_message`

---

## 5. 回写策略

### 5.1 GENERATE

成功回包后：

1. 题目 `PROCESSING/DISPATCHING -> REVIEW_PENDING`
2. AI 生成答案写入题目
3. `lastReviewMode = GENERATE`
4. task 标记成功

### 5.2 VALIDATE

成功回包后：

1. 题目 `PROCESSING/DISPATCHING -> REVIEW_PENDING`
2. 原答案不覆盖
3. 新增一条校验记录，保存：
   - `task_id`
   - 原答案快照
   - AI 建议答案
   - 校验结论
   - 原因
4. `lastReviewMode = VALIDATE`
5. task 标记成功

### 5.3 失败回包

1. 题目进入 `PROCESS_FAILED`
2. task 标记 `FAILED`
3. `failure_reason` 落到 task

---

## 6. 审核契约

### 6.1 GENERATE

- `APPLY_AI`
- `REJECT`

### 6.2 VALIDATE

- `KEEP_ORIGINAL`
- `APPLY_AI`
- `REJECT`

其中 `VALIDATE` 的审核对象，不是题目主表本身，而是本次任务产出的校验记录。

---

## 7. 当前异步实现约束

1. 一个题目同一时刻只能有一个有效 task。
2. `question.process_status` 是业务态，不再单独承担“消息是否真的发出、回包是否可信”的职责。
3. outbox 负责“发出去”，inbox 负责“别重复吃”，task 负责“这次业务到底是谁、走到哪了”。
4. 回包消费先过 inbox，再按 `taskId` 找 task，再决定如何改题目和校验记录。

---

## 8. 本地开发模式

1. `feature.mq.enabled=false`
   - 不连接 MQ
2. `feature.mq.mock-enabled=true`
   - 本地模拟 AI 成功回写，便于联调业务态
