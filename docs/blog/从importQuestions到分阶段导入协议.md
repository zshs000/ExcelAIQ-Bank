# 从 `importQuestions()` 到分阶段导入协议：复杂度该放在哪里？

之前我写过一篇文章，标题是：

> **《Excel 导入为什么要先校验再解析：一次复盘后的重新理解》**

上一篇文章讨论的重点，是我为什么坚持把 Excel 导入拆成两个阶段：

- 先上传
- 先做充分的前置校验
- 校验通过后，再让用户自己决定是否继续解析导入

它解决的是一个非常现实的问题：

> **用户可以处理的错误，应该尽可能提前暴露，并且明确告诉用户。**

这个认知来自我在医保局实习时接触到的真实场景。  
基层村干部导入一批人员信息表，可能只是某个格子里多了一个空格，整个 Excel 就直接失败。更离谱的是，系统给出的往往不是明确错误，而是一句“导入失败”，再加上一大段 Java 运行报错。

对开发者来说，那是异常栈。  
对基层用户来说，那就是天书。

他们不知道哪一行错了，不知道为什么错，也不知道下一步该怎么改。最后只能带着文件，跑很远来县城办公室，让我们现场帮忙排查。很多时候，问题甚至只是某个单元格里多了一个空格。

所以后来在我自己的系统里，我没有继续把 Excel 导入做成“一上传就直接正式处理”的黑箱流程，而是拆成了两个阶段：

- 不再把 Excel 上传后直接丢进正式导入
- 先做一次独立的、充分的前置校验
- 校验内容包括：模板、格式、空值、空格、字段规则等
- 校验失败时，不进入正式导入，而是直接告诉用户明确错误
- 用户在校验通过后，再自己决定是否继续解析

这个设计到现在我依然认为是对的。  
因为它做的不只是“多一次流式解析”，它实际上是在贯彻一条我越来越坚定的原则：

> **把控制权交给用户。**

但问题没有停在这里。

“先校验再解析”解决的是**导入前半段**的问题；顺着链路继续往下看，后半段其实还有一个更深的结构性问题。

## 一、看起来用了流式解析，实际上导入还是一次性的

当时正式导入链路大致是这样的：

1. 从 OSS 取回 Excel 文件
2. 用 EasyExcel 流式解析
3. 把每一行解析结果收集到 `List<QuestionDTO>`
4. 再一次性调用：

```java
importQuestions(List<QuestionDTO> questions)
```

乍一看，这条链路似乎已经挺像样了：

- 用了 EasyExcel
- 是流式解析
- 没有把整个 Excel 一次性读进内存

但问题在于：

> **解析阶段是流式的，不代表导入阶段也是流式的。**

所有解析出来的 DTO，最终还是会被全部聚合进一个 `List`，然后作为一次 RPC 请求，一口气扔给题库服务。

所以这条链路本质上还是：

> **前面看起来在流式，后面又重新变回了一次性黑箱导入。**

而且，Excel 文件本身的大小并不能真实反映这条链路的资源开销。

很多人看到一个 `xlsx` 文件只有几 MB，就会下意识觉得它不算大。  
但 `xlsx` 本身就是压缩格式。磁盘上的文件小，不代表它在内存里也小。

一旦你把它解析成：

- Java 对象
- 多个字符串字段
- `List<QuestionDTO>`
- JSON 序列化请求体
- RPC 传输数据

最终内存中的体积、序列化后的体积，往往比原文件本身还要大得多。

也就是说：

> Excel 文件小，不代表导入链路轻。  
> `xlsx` 是压缩后的结果，真正把它解析成对象以后，内存中的 DTO 列表、JSON 请求体、RPC 负载，可能比文件本身膨胀得多。

所以当时那条链路虽然已经不再是“整文件一次性读入内存”，但它仍然存在一个本质问题：

> **导入阶段的复杂度，依旧被塞进了一个 `importQuestions()` 的大黑箱里。**

## 二、真正的矛盾，不是“要不要流式”，而是“分批传输”和“最终原子性”之间的矛盾

当我把问题看到这里时，最自然的下一步就是：

既然解析可以流式，导入能不能也别一次性？

比如这样：

- 解析 100 条
- 推 100 条
- 再解析 100 条
- 再推 100 条

这看起来非常合理，也确实能缓解：

- 内存压力
- RPC 单次请求体过大
- 一次性大黑箱调用的问题

但新的问题马上就出现了。

如果我真的这样做，并且让题库服务把每一批数据都直接写进正式表，那么一旦某一批失败，就会出现：

- 前几批已经成功写入正式表
- 后几批还没写
- 整个导入结果只完成了一半

这时候，系统就必须正面回答一个问题：

> **我到底接不接受“部分成功”？**

在这里，我最后给出的答案是：

> **不接受。**

这才是这篇文章真正的核心矛盾：

# **工程上，我不想继续保留一个一次性的大黑箱导入。**  
但业务上，我又仍然坚持最终结果必须是原子的。

也就是说，我真正面对的不是“流式还是非流式”的问题，而是：

# **分批传输** 和 **最终原子提交** 之间的矛盾。

## 三、为什么我最后仍然坚持“最终原子性”

先说清楚一点：

我并不是迷信事务，也不是单纯觉得“全成全败”听起来更高级。  
我最后坚持原子性，是因为这条导入链路的业务边界已经越来越清楚了：

> **用户可以处理的问题，应该在前置校验阶段暴露；  
> 一旦进入正式导入阶段，就不应该再把系统内部的中间态暴露给用户。**

这一点和我在医保局实习时形成的认知，其实是一脉相承的。

当年真正让我不爽的，不是“整批失败”这件事本身。  
我不接受的是：

- 明明是用户可以修改的错误
- 系统却不提前告诉他
- 非要等到正式导入时才炸
- 炸了以后还不给明确错误信息

所以后来在自己的系统里，我做的事情，本质上不是否定原子性，而是：

> **把该前置的错误尽量前置，把该交给用户的控制权尽量交给用户。**

在这个前提下，正式导入阶段的语义就会变得非常清楚：

- 如果是模板、格式、空值、空格、字段规则问题  
  → 这些应该在前置校验阶段被拦下来
- 如果前置校验已经通过，正式导入阶段还出错  
  → 那大概率已经不是“用户数据有问题”，而是系统问题、依赖问题、状态问题

既然如此，我就不想在正式导入阶段再给出一种模糊语义：

- 成功 83 条
- 失败 17 条

这看似灵活，实际上非常脏。  
因为这 17 条到底是业务错误、系统错误、网络错误还是下游错误，用户根本分不清。

所以这里的结论非常明确：

> **前置阶段要尽量友好、尽量细致、尽量把错误说清楚；  
> 正式提交阶段则应该保持原子性。**

也就是：

- 用户问题，前置解决
- 系统提交，一次完成

## 四、如果既想分批传输，又想最终原子，那就必须引入中间态

到这里，问题其实已经很明确了。

我同时想要两件事：

### 第一，不继续保留 `importQuestions(List<QuestionDTO>)` 这种一次性黑箱调用

因为它会把：

- 内存压力
- 序列化压力
- 传输压力
- 大请求风险

全部堆到一次请求里。

### 第二，不让每一批数据直接进入正式表

因为那会破坏最终原子性。

那怎么办？

答案只能是：

> **在分批传输和正式提交之间，加一个中间承接层。**

也就是：

- 解析时可以按 chunk 一批一批推送
- 但这些 chunk 不直接进入正式题目表
- 而是先进入临时表
- 等所有 chunk 都接收完成，再统一 commit 到正式表

这样，两个目标就同时保住了：

- **过程是分批的**
- **结果是原子的**

所以后来的方案，自然就落在了这里：

# **Batch + Chunk + 临时表 + 最终 Commit**

## 五、具体实现：四个接口，四个阶段

为了把这个过程真正建模清楚，我最终没有再保留一个 `importQuestions()` 大接口，而是拆成了四个动作：

### 1. `createImportBatch`

创建一个导入批次。

它的职责很单纯：

- 生成批次号
- 记录这次导入属于哪个文件、哪个用户
- 初始化状态为 `APPENDING`

### 2. `appendImportChunk`

按批次追加一个 chunk。

它负责：

- 接收这一小批解析结果
- 写入临时表
- 更新这个 batch 已经接收了多少 chunk、多少行

这里有一个非常关键的点：

> **“接收一个 chunk”本身必须是原子的。**

也就是说：

- temp 表写入
- batch 计数更新

这两个动作必须在同一个事务里。  
要么都成功，要么都失败。

否则就会出现那种非常恶心的中间态：

- chunk 的行已经落到临时表了
- 但 batch 的计数没更新
- 下一次重试时，系统又以为这个 chunk 没完整接收

### 3. `finishImportBatch`

声明：“我已经传完了。”

这个动作不负责正式入库，它负责的是**封口**：

- 我预期应该有多少个 chunk
- 一共应该有多少行
- 当前实际接收到的是不是一致

如果不一致，说明这次传输链路本身就有问题，batch 应该直接失败，而不是继续往下走。

### 4. `commitImportBatch`

最终正式提交。

它的职责是：

- 从临时表按顺序取出这次 batch 的所有行
- 组装正式题目对象
- 一次性写入正式表
- 把 batch 标记为 `COMMITTED`

这个阶段，才是真正意义上的“正式导入”。

所以整个协议就非常清晰：

- 先 `create`
- 再 `append` 若干次
- 再 `finish`
- 最后 `commit`

这已经不是普通 CRUD 了，而是一套有明显生命周期的导入协议。

## 六、复杂度并没有消失，它只是从 `importQuestions()` 里被逼了出来

走到这里，整件事就很清楚了：

以前那个 `importQuestions(List<QuestionDTO>)` 看起来很简单。  
但它只是把复杂度藏起来了。

真正被藏起来的复杂度包括：

- 大请求传输
- 大对象聚合
- 分批重试
- 幂等判断
- 中间状态管理
- 最终提交边界
- 失败恢复
- 超时清理

这些复杂度本来就存在。  
只是以前它们被强行压扁进了一个“看似简单”的接口里。

而现在，这些复杂度被重新放回了它们真正该在的位置：

- **batch**
- **chunk**
- **临时表**
- **事务边界**
- **幂等校验**
- **状态推进**
- **清理任务**

换句话说：

> 这次重构并不是把系统变简单了，  
> 而是不再假装它本来就很简单。

也正是在这里，我越来越认同那句话：

> **复杂度不会消失，它只会转移到它该在的位置。**

这一次，它不该继续藏在一个黑箱式的 `importQuestions()` 里；  
它应该进入导入协议本身，进入 batch、chunk、临时表、幂等和事务边界这些真正属于它的地方。

## 七、当然，这个方案也有成本，它不是无脑正确的

这一点必须说清楚。

这套方案并不是“永远更高级”，它只是更符合我当前的业务约束。

它引入了新的复杂度，比如：

- 需要 batch 状态管理
- 需要 chunk 级幂等
- 需要临时表
- 需要清理任务
- 需要处理中途失败、超时中止
- 需要保证 `append` 的事务边界
- 需要在 `finish` / `commit` 之间维护一致性

这些都是真复杂度，不是假复杂度。

所以这篇文章如果只写“这套方案多优雅”，那是不诚实的。  
更准确的说法应该是：

> 我并没有消灭复杂度，  
> 我只是接受了这些复杂度本来就属于这条导入链路，然后把它们放到了业务协议内部，而不是继续甩给一个黑箱式的 `importQuestions()`。

另外，这套方案也没有回避另一个很现实的问题：

> **最终 `commit` 阶段仍然可能是一个大事务。**

因为在所有 chunk 都已经进入临时表之后，正式提交阶段依然需要把整批数据一次性写入正式表。只要坚持“最终结果必须原子”这个语义，这里的大事务风险就不会凭空消失。

这一点我不是没看到，而是明确接受了。

原因也很直接：  
如果为了规避大事务，把正式表写入改成多事务分批提交，那么导入语义就会从“最终原子”滑向“部分成功”。而这恰恰是我前面一直在避免的结果。

所以这里的取舍其实很明确：

- **过程可以分批**
- **中间可以承接到临时表**
- **但最终提交仍然保持原子**

这不是在否认大事务风险，而是在当前业务前提下，先把这项风险视为方案成本，并通过后续测试去确认它的可接受范围。

当然，这个选择也有边界。

Excel 单个工作表理论上最多可以到一百万行，但我的系统并不是那种超大规模数据导入系统。哪怕抛开“这是一个练习项目”的初心不谈，单从题库导入这个业务场景本身来看，站在用户角度，一次导入上万条题目基本已经是很高的量级了，再往上走，本身就不是常见使用方式。

所以这条链路的现实前提其实是：

1. **它首先是一个练习和沉淀架构思考的项目**
2. **但方案设计不能只停留在练习，而要尽量贴近真实业务约束**
3. **放到题库导入这个具体场景里，用户一次导入的数据量通常有很明确的现实上限**

在这个前提下，我愿意先接受“最终原子提交会带来一个可控的大事务风险”，而不是为了提前规避一个未必会真的发生的极端规模问题，直接把业务语义改坏。

后面我会专门对这一点做测试：  
随着临时表数据量逐步增加，观察最终 `commit` 阶段的一次性原子提交到底在什么数量级下开始变得明显吃力，从而为这个方案补上一个更明确的工程边界。

## 八、为什么我最后还是选择了它

成本和边界说清楚以后，剩下的问题就很简单了：为什么即使如此，我还是选择了它？

### 1. 用户可以处理的问题，应该前置校验

模板、空值、空格、格式、字段规则，尽量在正式导入前告诉用户。

### 2. 用户应该拥有控制权

不是一上传就黑箱处理，而是先校验，再由用户决定是否继续。

### 3. 正式导入阶段仍然坚持最终原子性

一旦前置校验通过，正式提交就不应该暴露半成功半失败的中间态。

### 4. 导入过程本身不能继续维持一次性大黑箱接口

否则前面所谓的流式解析，最终还是会在 `List<QuestionDTO>` 和 RPC 大请求里被抵消掉。

所以对我来说，这套方案真正解决的不是“怎么把导入流程搞得更复杂”，而是：

> **在不放弃最终原子语义的前提下，把原来被黑箱接口吞掉的导入过程拆出来。**

在这个前提下，  
“分批传输 + 临时表承接 + 最终原子提交”就不再是一个炫技式设计，而是一个自然的结果。

## 九、结语

那篇“先校验再解析”的文章，解决的是导入链路的前半段：  
为什么要把用户错误前置、为什么要把控制权交给用户。

而这次继续往下重构，解决的是后半段：

> **在保留最终原子性的前提下，怎么把导入过程从一个看似简单的黑箱接口里拆出来。**

这件事让我对导入系统的理解更完整了一步。

`importQuestions()` 不是消灭了复杂度，只是把复杂度藏起来了。  
真正的重构，不是让系统“看起来更简单”，而是让复杂度回到它本来就该在的地方。

这一次，它最终回去的位置是：

- batch
- chunk
- 临时表
- 幂等
- 状态推进
- 事务边界
- 清理任务

这些地方一点都不轻松，甚至可以说一点都不便宜。  
但至少，这一次，它们被放回了正确的位置。
