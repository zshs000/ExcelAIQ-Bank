# 从单体全局异常处理到微服务错误契约：为什么不能吞下游错误

## 1. 这篇文章想解决什么问题

我以前一直觉得，自己已经把“统一异常处理”这件事学明白了。

在单体项目里，这套东西很顺：

- 业务里发现错误，直接抛 `BizException`
- `GlobalExceptionHandler` 统一捕获
- 最后返回统一格式的 `Response.fail(...)`

代码干净，接口统一，逻辑也很好推理。

所以很容易产生一种错觉：

> 统一异常处理，我已经会了。

真正把项目拆成微服务以后，我才发现，这件事根本没有结束。

单体里那套模型并没有失效，只是它不再完整了。因为一旦跨服务，异常不能直接穿透网络，系统就会多出一层全新的问题：

- 下游错误要怎么表达
- 上游要怎么接收
- 谁负责把失败 DTO 恢复成当前服务内部的异常
- 哪一层可以做业务语义翻译
- 哪一层绝对不能吞错

也就是从这里开始，我才真正把“统一异常处理”理解成两层：

- 单体内的异常收口
- 分布式边界上的错误契约治理

这篇文章就想把这两层一次讲透。

## 2. 单体里那套异常处理，到底解决了什么

先把最基础的那一层说清楚。

在单体项目里，大家通常都会做这样一套东西：

1. 定义统一业务异常，比如 `BizException`
2. 业务代码里一旦发现错误，直接 `throw new BizException(...)`
3. 用 `GlobalExceptionHandler` 统一捕获
4. 最终对外返回统一结构的失败响应，比如 `Response.fail(errorCode, message)`

这套设计为什么这么常见？

因为它把两件事分开了：

- 业务上哪里出错了
- 接口最终要返回什么格式

业务层只需要关心：这里是不是已经失败了。如果失败，就直接抛异常，终止流程。

至于：

- 错误码如何组织
- 错误消息如何返回
- 参数校验失败如何统一格式
- 未知异常如何兜底

这些都交给统一异常处理器去做。

所以单体里的一个重要认识是：

- 异常是进程内控制流
- `Response` 是接口出参协议

这个区分非常重要。

很多人只记住了“要有全局异常处理器”，却没有真正记住这句话。结果一到微服务就开始乱了。

## 3. 问题的关键转折：微服务里，异常不能直接跨网络

在单体里，异常可以这样一路往上抛：

- service 抛异常
- controller 不处理
- `GlobalExceptionHandler` 接住
- 返回统一失败响应

因为大家都在同一个 JVM 里。

但微服务不是这样。

当服务 A 调服务 B 时，服务 B 内部抛出的 Java 异常，不可能直接穿透网络，跑到服务 A 的调用栈里。跨服务真正能传递的，只有网络协议能表达的内容。

如果项目统一使用 `Response<T>` 作为接口返回包装，那么跨服务真正能传过去的，就是：

- 成功：`Response.success(data)`
- 失败：`Response.fail(errorCode, message)`

也就是说，服务 B 内部抛出的异常，到服务边界时必须被翻译成失败 DTO。

所以从这里开始，事情就变成了：

- 服务内部仍然是异常模型
- 服务边界必须落成 DTO 模型

这不是多此一举，而是分布式系统的客观事实。

## 4. 一条完整的正确链路，到底长什么样

这一点如果不讲清楚，后面所有“别吞错”的结论都会显得很散。

一条正确的分布式错误链路，通常应该是这样的。

### 4.1 服务提供方

服务提供方内部：

- 业务失败时，抛 `BizException`
- `GlobalExceptionHandler` 统一捕获
- 对外返回 `Response.fail(errorCode, message)`

也就是说：

- 内部世界是异常
- 出口世界是 DTO

### 4.2 服务消费方

服务消费方拿到下游返回的 `Response` 后，不能直接把这个 DTO 原封不动扔给业务层。

它应该先在 RPC 适配层做一层协议恢复：

- `response == null` -> 抛异常
- `!response.isSuccess()` -> 抛异常，并保留下游 `errorCode / message`
- `data` 缺失但按契约本应存在 -> 抛异常
- 只有成功且数据完整时，才把结果返回给业务层

于是，这条链路就闭环了：

- 服务提供方内部抛异常
- 服务边界转成失败 DTO
- 服务消费方的 RPC 适配层再恢复成异常
- 当前服务内部继续走统一异常流

这才是一个完整的模型。

## 5. 我后来真正意识到：RPC 适配层不是“顺手封个 Feign”

很多项目里都有一个叫 `xxxRpcService`、`xxxRemoteService`、`xxxClientAdapter` 的类。

一开始我也把它理解得很轻：

- 封一层 Feign 调用
- 少让业务层直接写 HTTP/Feign 代码
- 看起来整洁一点

后来我才发现，这一层根本不是可有可无的包装层。

它其实承担着非常关键的职责：

- 处理远程协议细节
- 统一校验下游响应
- 恢复下游错误语义
- 做有限重试
- 决定哪些场景可以特判翻译
- 把业务层和远程调用细节隔开

换句话说：

> 业务层面向业务语义，RPC 层面向远程协议语义。

这句话特别重要。

如果远程协议细节一路泄漏到业务层，比如业务层到处都在写：

- `response == null`
- `response.isSuccess()`
- `response.getErrorCode()`
- `FeignException`
- `RetryableException`

那说明边界层没有收住。

而一旦边界层没收住，业务层就会开始做一堆本不该它做的事情：

- 猜测下游为什么失败
- 重复写空响应判断
- 把远程失败改写成通用业务失败
- 把真实错误吞掉再重新发明一层模糊语义

这时候系统就会慢慢变得越来越难推理。

所以我现在的理解是：

RPC 适配层不是“可选优化”，而是一层非常有价值的边界层。它的职责不是让代码看起来更整洁，而是让远程调用规则集中、稳定、可维护。

## 6. 真正的坑，不是“下游返回失败”，而是“错误语义被吞了”

很多人第一次听到这个问题，会觉得不理解：

“下游不是已经返回 `Response.fail(...)` 了吗？那不就已经是失败了吗？还有什么问题？”

问题在于，上游拿到这个失败 DTO 之后，怎么处理。

最糟糕的做法就是：

- 下游已经返回失败 DTO
- 上游 RPC 适配层不抛异常
- 也不保留 `errorCode / message`
- 而是直接 `return null`

这就是错误语义被吞掉了。

比如下游明明已经告诉你：

- 文件类型错误
- 参数非法
- 访问凭证生成失败
- 服务响应为空

结果 RPC 适配层转头只告诉业务层一句：

- 我给你个 `null`

问题就从这一刻开始失控。

因为业务层根本分不清：

- 这是真没数据
- 还是下游服务挂了
- 还是下游明确返回了业务失败
- 还是响应为空
- 还是数据字段缺失

于是业务层就只能开始猜。

而一旦开始猜，整个系统的可推理性就下降了：

- 错误码不准确
- 日志不可信
- 调用链断裂
- 重试决策容易失真
- 补偿逻辑容易误判
- 前端拿到的也可能是假语义

所以“不能吞错误”不是一句抽象道德要求，而是一个很具体的分层原则。

## 7. 另一种常见坑：双层 success 语义打架

除了把错误吞成 `null`，还有一种问题也很常见。

那就是同一个接口里，同时保留了两套成败语义。

典型形式是：

- 外层 `Response.success(...)`
- 内层 data 又带一个 `success=false`、`status=FAILED` 或类似字段

看起来它似乎在表达：

- 请求调通了
- 但业务处理失败了

可问题是，调用方到底该信哪一层？

如果前端只看外层：

- 会误以为成功

如果前端只看内层：

- 那外层的 `Response.success` 就变成了噪音

于是一个接口同时维护两套成功/失败语义，最后谁都说不清楚。

所以我后来给自己定了另一条规则：

> 一个接口最好只保留一套成败语义。

要么：

- 外层 success / fail 就是唯一语义

要么：

- 外层永远只表示“请求已受理”，内层结果对象才是唯一业务语义

但绝对不能两套都想要，却没有明确边界。

## 8. 项目里让我真正吃透这件事的几个例子

### 8.1 `auth -> user`：正例

这条链路相对规范。

`user` 服务内部失败时：

- 抛 `BizException`
- 对外由全局异常处理器转成 `Response.fail`

`auth` 调它时，RPC 层做的是：

- `response == null` -> 抛异常
- `!response.isSuccess()` -> 抛异常，并保留下游 `errorCode / message`
- 只有极少数明确业务场景，比如“用户不存在”，才翻译成 `null`

这就是非常标准的模式：

- 默认保真
- 只有明确约定过的业务语义，才允许特判翻译

### 8.2 `user -> oss`：反例

这条链路以前的问题是：

- OSS 如果返回失败 DTO
- `user` 的 RPC 适配层不抛异常
- 而是直接 `return null`

于是上传头像/背景图时，业务层只能把这个空值改写成一个通用的“上传失败”。

读取资料生成图片访问链接时，甚至可能把“OSS 签名失败”伪装成“用户没有图片”。

这就是非常典型的错误语义吞没。

### 8.3 `excel-parser -> oss`：反例

这条线以前也有两个问题：

- 上传 Excel 失败时，RPC 层返回 `null`
- 获取下载访问凭证时，重试三次之后还是失败，也返回 `null`

于是业务层：

- 上传失败被统一压成通用业务码
- 下载失败再自己改写成统一提示

从“用户有没有看到失败”这个角度看，好像也不是完全错。

但从系统角度看，下游的真实失败原因已经丢了。

### 8.4 `excel-parser -> question-bank`：双层 success 语义冲突

还有一条更有代表性的链路，是 `excel-parser -> question-bank`。

下游导入失败后：

- 先被包装成一个 `success=false` 的结果 DTO
- 外层调用最后却仍然 `Response.success(...)` 返回给前端

这就同时踩中了两个坑：

- 错误没有回到统一异常语义
- 外层和内层成功语义打架

所以我后来越来越确定：

这类问题的本质，通常都不是某一行代码写错了，而是服务边界上的契约治理没有收口。

## 9. 什么时候可以翻译，什么时候绝对不能乱翻译

这里要特别讲清楚，不然很容易又走到另一个极端：

“是不是所有下游失败都必须一律抛异常，绝不能翻译？”

也不是。

关键不在于能不能翻译，而在于这个翻译是不是一个明确、稳定、事先约定好的业务语义。

### 9.1 可以翻译的情况

比如：

- 下游返回“用户不存在”
- 上游登录流程把它翻译成“查无此人”
- 然后返回 `null`

这就是合理的。

因为这里的 `null` 有明确业务含义：

- 它不是“服务挂了”
- 也不是“不知道发生了什么”
- 而是“这个用户确实不存在”

### 9.2 不能乱翻译的情况

以下这些情况，默认都不该翻译成 `null`：

- 下游响应为空
- Feign 调用失败
- 下游返回 `success=false`
- 下游返回数据缺失，但按契约本应存在
- 下游已经给出了明确错误码和错误消息

因为这些都不是正常业务分支，而是错误。

它们要么应该被透传，要么应该被显式包装成新的异常，但绝不能直接被吞掉。

所以我后来给自己定了一条很实用的规则：

> RPC 适配层默认保真，不负责瞎翻译。只有领域上明确成立的场景，才允许把失败语义翻译成别的语义。

## 10. 一套适合大多数项目复用的规范

如果把这些经验收口成一套可复用规则，我现在会这么定。

### 10.1 服务提供方

服务提供方内部：

- 业务失败统一抛 `BizException`
- 参数错误可以用 `BizException`，也可以用 `IllegalArgumentException`，但最终都要统一收口
- controller / 全局异常处理器统一转成 `Response.fail`

也就是说：

- 内部世界是异常模型
- 对外世界是 DTO 协议模型

### 10.2 服务消费方的 RPC 适配层

RPC 适配层默认应该这样处理：

- `response == null` -> 抛异常
- `!response.isSuccess()` -> 抛异常，并保留下游 `errorCode / message`
- `data` 缺失但按契约必须存在 -> 抛异常
- 有限重试逻辑也放在这一层
- 只有明确业务语义场景，才允许做翻译

这一步很关键，因为它决定了：

- 下游错误语义能不能被完整带回来
- 本服务内部还能不能继续走统一异常流
- 业务层会不会被远程协议细节污染

### 10.3 业务层

业务层不要负责猜测远程调用失败的原因。

业务层应该只负责：

- 本地事务与业务编排
- 状态写回
- 补偿标记
- 领域语义翻译
- 最终对外结果组织

但不应该在收到一个 `null` 之后开始脑补：

- 是不是下游挂了
- 是不是业务失败
- 是不是没数据

因为一旦业务层开始猜，说明边界层已经失职了。

### 10.4 接口设计

一个接口只保留一套成败语义。

不要：

- 外层 `Response.success`
- 内层再放 `success=false`

如果确实要表达“请求已受理，但业务结果异步完成”，那就从设计一开始把语义定死：

- 外层表示“请求受理状态”
- 内层表示“业务处理状态”

而不是让两套语义后期在一个接口里打架。

## 11. 为什么这件事在分布式里尤其重要

在单体里，吞错最多只是让排查困难一些。

但在分布式里，吞错的代价会被放大很多。因为错误语义会直接影响：

- 重试策略
- 补偿策略
- 状态机判断
- 告警准确性
- 调用链日志
- 上游系统的分支行为

举个最现实的例子。

如果下游已经明确返回“访问凭证生成失败”，而你上游把它吞成了 `null`，那上游可能就会误判成：

- 没有图片
- 文件不存在
- 可以继续走兜底分支

甚至会进一步写出错误的状态、错误的日志、错误的提示文案。

所以这件事从来不是代码洁癖。

它的本质是：

> 你有没有把分布式边界上的错误契约定义清楚。

## 12. 最后的反思

很多人对“统一异常处理”的理解，其实只完成了一半。

大家会写：

- `BizException`
- `GlobalExceptionHandler`
- `Response<T>`

然后就觉得自己已经掌握了这件事。

但真正把项目做成微服务以后，你才会发现：

- 服务边界会切断异常传播
- DTO 会成为新的错误载体
- RPC 适配层会成为新的契约入口
- 一旦这里乱翻译，整条链路都会失真

所以完整的模型应该是：

- 单体内部：异常流统一
- 服务边界：异常落成 DTO
- 服务消费方：DTO 恢复为异常
- 只有明确业务场景才允许特判翻译

也就是说，全局异常处理不是终点，而只是第一层。

到了微服务阶段，你还必须补上“错误契约治理”这一层。否则你得到的只会是一个表面统一、实际混乱的系统：

- 每个服务看起来都有 `Response<T>`
- 每个服务看起来都有 `GlobalExceptionHandler`
- 但真正跨服务时，错误语义却被吃光了

这才是最危险的状态。

## 13. 一段我现在会直接背出来的话

全局异常处理解决的是单体内部的错误收口问题：服务内部抛异常，边界统一转失败响应。

但到了微服务里，异常不能直接跨网络传播，所以错误必须先在服务提供方边界转换成失败 DTO，再在服务消费方的 RPC 适配层恢复成异常。

因此，真正需要统一的不只是 `GlobalExceptionHandler`，还包括跨服务错误契约，尤其是 RPC 适配层这一层：

- 下游返回什么
- 上游如何接收
- 哪些错误必须原样保留
- 哪些场景才允许做业务语义翻译

RPC 层默认应该保真，而不是吞掉下游错误，把它变成 `null`、通用失败，或者双层 success 这种混乱语义。

因为在分布式系统里，错误语义一旦被吞掉，丢掉的不只是一个错误码，而是整条链路的可推理性。
