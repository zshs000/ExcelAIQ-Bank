# 异步 AI 链路重构复盘：从拆状态到 task / outbox / inbox

## 1. 这次重构到底在解决什么

这次改造的起点，其实不是一上来就想到 `outbox`。

最开始暴露出来的问题，是题目进入 AI 处理链路后，题目主表里的状态语义不够清楚。原先一个 `WAITING -> PROCESSING` 的流转太粗了，系统里缺少“已经进入发送流程，但还没确认真正发出去”的中间阶段。

所以第一步我先做了状态拆分，引入了：

- `WAITING`
- `DISPATCHING`
- `PROCESSING`
- `REVIEW_PENDING`
- `COMPLETED`
- `PROCESS_FAILED`

其中 `DISPATCHING` 的意义很明确：

**题目已经进入可靠派发流程，但消息还没有被确认送到 AI 处理链路。**

这一步很重要，因为它先把业务状态语义理顺了。

但问题也恰恰出在这里：

**只拆状态，不足以解决双写一致性问题。**

---

## 2. 为什么加了 `DISPATCHING` 还是不够

一开始我以为，把题目从 `WAITING` 改成 `DISPATCHING`，然后再去发 MQ，已经算是把链路收紧了。

后来继续往下看，才发现核心矛盾根本不在状态名字，而在于：

- 改题目状态，是本地数据库事务
- 发 RocketMQ，是外部消息系统调用

这两个动作天然不在同一个事务里。

也就是说，即使我把状态拆得再细，依然会遇到下面这种问题：

1. 本地数据库已经提交，题目进入 `DISPATCHING`
2. 但消息还没真正发出去，或者发送结果不确定

或者反过来：

1. MQ 实际已经收到了消息
2. 但本地后续状态同步没有完全推进

这就是典型的双写问题。

所以这一轮真正的顿悟不是“再多加一个状态”，而是：

**必须把原本无法纳入本地事务的“发送 MQ”动作，先翻译成一条本地可持久化的事实。**

这就是 `outbox` 的核心思想。

---

## 3. 最终落下来的三张过程表

这次最后收口成了三张过程表：

1. `question_process_task`
2. `question_outbox_event`
3. `question_callback_inbox`

这三张表不是为了“显得架构更复杂”，而是为了把原来都压在题目主表上的职责拆开。

### 3.1 `question_process_task`

`task` 表记录的是：

**这次异步处理任务本身是谁。**

它至少包含：

- `id`
- `question_id`
- `mode`
- `attempt_no`
- `task_status`
- `callback_key`
- `failure_reason`

它解决的问题是：

- 同一道题的一次具体异步处理，必须有独立身份
- 回包不能只靠 `questionId` 猜，而要认“这次任务”
- 后续如果有业务重试，也应该形成新的 `attemptNo`

所以现在的判断标准变成了：

**异步链路的真实身份以 `task` 为准，题目主表只是业务外观。**

### 3.2 `question_outbox_event`

`outbox` 表记录的是：

**这条任务对应的消息发送账本。**

它至少包含：

- `id`
- `task_id`
- `event_status`
- `dispatch_retry_count`
- `created_time`
- `updated_time`

它解决的问题是：

- 题目已经决定进入 AI 链路，这个“发送意图”必须先在本地库落下来
- 即使这一次没有真正发出去，账本还在，后面还能继续扫描重发

当前 outbox 状态比较简单：

- `NEW`：刚落库，还没发
- `SENT`：已经成功派发
- `RETRYABLE`：这次发送失败，后面还可以继续扫

这里最关键的一点是：

**outbox 不是业务任务本身，它只是“这条任务的发送账本”。**

### 3.3 `question_callback_inbox`

`inbox` 表记录的是：

**回包消费幂等账本。**

它至少包含：

- `id`
- `callback_key`
- `task_id`
- `consume_status`
- `created_time`
- `updated_time`

它解决的问题是：

- AI 结果回包重复投递时，不能重复推进题目状态
- 回包消费要先建立 inbox 记录，再处理业务，再把 inbox 从 `RECEIVED` 推到 `PROCESSED`

所以 `inbox` 的本质就是：

**接收侧幂等控制。**

---

## 4. 这三张表分别在链路里负责什么

如果用一句话概括：

- `task` 负责定义“这次任务是谁”
- `outbox` 负责定义“这条消息发出去没有”
- `inbox` 负责定义“这条回包吃过没有”

也可以换一种更工程化的表达：

- `task`：业务任务身份和生命周期
- `outbox`：发送账本与发送重试
- `inbox`：回包幂等与重复消费拦截

它们三个拆开后，题目主表就不用再同时承担：

- 业务展示状态
- 消息是否发出
- 回包是否可信
- 回包是否重复

这也是这次重构真正的价值。

## 5. 当前链路现在是怎么跑的

### 5.1 请求提交阶段

用户在前端发起“发送到 AI”后，主链路不再同步发 MQ，而是只做本地准备：

1. 筛选题目
2. 校验当前用户权限
3. 按 `GENERATE / VALIDATE` 模式分流
4. 题目 `WAITING -> DISPATCHING`
5. 创建 `task`
6. 创建 `outbox`
7. 直接返回“已提交异步派发”

这里的核心变化是：

**请求线程不再直接承担 MQ 发送动作，只负责把发送意图安全落库。**

### 5.2 后台派发阶段

后台由一个简单的 Poller 扫描 outbox：

1. 扫 `NEW`
2. 扫 `RETRYABLE`
3. 找到对应 `task`
4. 找到对应 `question`
5. 组装消息
6. 真正调用 `rocketMQTemplate.syncSend(...)`
7. 成功后推进：
   - outbox: `NEW/RETRYABLE -> SENT`
   - task: `PENDING_DISPATCH -> DISPATCHED`
   - question: `DISPATCHING -> PROCESSING`

这一层完成以后，消息才算真正进入 AI 处理链路。

### 5.3 AI 回包阶段

AI 返回结果后，题目服务先不急着改题目，而是先过两层校验：

1. `inbox` 幂等校验
2. `taskId / questionId / attemptNo / active task` 身份校验

只有通过后，才继续推进业务。

成功回包时：

- `GENERATE`：可以直接写入 AI 结果，再推进到 `REVIEW_PENDING`
- `VALIDATE`：不直接改原答案，而是新增一条带 `taskId` 的校验记录，再推进到 `REVIEW_PENDING`

失败回包时：

- 题目推进到 `PROCESS_FAILED`
- `task` 记录失败原因

### 5.4 人工审核阶段

到 `REVIEW_PENDING` 后，才进入人工审核。

当前已经拆成两套语义：

- `GENERATE`
  - `APPLY_AI`
  - `REJECT`

- `VALIDATE`
  - `KEEP_ORIGINAL`
  - `APPLY_AI`
  - `REJECT`

也就是说：

- 生成模式更像“AI 先给了一个完整候选结果”
- 校验模式更像“AI 给建议，最后由人工决定是否覆盖原答案”

## 6. 题目状态推进为什么也能说是 CAS

这次链路里，题目主状态推进大量用了数据库条件更新。

例如：

```sql
update t_question_bank
set process_status = #{targetStatus}
where id = #{id}
  and process_status = #{expectedStatus}
```

它本质上就是：

- 先比较当前状态是不是预期旧值
- 只有一致，才允许改成目标状态

这就是数据库条件更新式 CAS。

所以这里虽然不是 JVM 原子类的 `compareAndSet()`，但思想完全一致：

**带前置条件地推进状态，而不是无脑覆盖。**

这能避免并发下：

- 老线程按旧状态误更新
- 重复回包重复推进
- 不同阶段互相覆盖

## 7. 这次最大的反思：不是所有问题都能靠“再加一个状态”解决

这次最让我印象深的地方就是：

一开始我已经意识到要“拆状态”，也确实加了 `DISPATCHING`。

但后来发现：

**状态拆分只能解决状态语义问题，解决不了本地数据库和外部消息系统天然不在同一事务里的事实。**

所以这次真正更进一步的地方，不是又加了几种状态，而是：

- 先承认流程是分阶段完成的
- 再承认外部动作无法强行塞进本地事务
- 最后把它翻译成一个本地可靠记录，再异步推进

从这个角度看：

- `DISPATCHING` 是中间状态
- `outbox` 也是中间状态
- `inbox` 本质上也是对消费阶段中间事实的持久化

这次重构再次验证了一点：

**只要流程不是一步原子完成的，就一定要显式承认中间态。**

---

## 8. 当前这版已经解决了什么

当前这版已经解决了几个核心问题：

1. 主请求线程不再同步发 MQ，用户提交不会被消息发送阻塞
2. “发送意图”先在本地事务中落地，双写问题被明显收紧
3. 异步链路的真实身份从题目状态切换到了 `task`
4. 回包消费不再只靠题目状态猜，而是 `inbox + task` 双重校验
5. `GENERATE / VALIDATE` 两种模式的业务语义已经基本拆开

这意味着：

**现在已经不是“靠 catch 回滚一下状态”的链路了，而是一条有身份、有账本、有幂等的可靠异步主干。**

---

## 9. 但这还只是第一版，不是终态

这次落地的是一个“最小可用”的可靠异步版本，还不是完整生产版。

当前明确还没补齐的点包括：

1. Poller 还只是单服务内的 `@Scheduled`
2. 扫描 SQL 还没有 `limit`
3. 多实例下还没有真正的 claim 机制或分布式锁保护
4. outbox 还没有最大重试次数、死信状态和错误明细
5. 还没有“长时间停留在处理中”的超时扫描与告警
6. 还没有用户侧的主动通知机制

所以更准确地说：

**现在已经把异步主干搭起来了，但运维补偿、集群并发控制和用户通知层还在后续演进范围内。**

---

## 10. 后续如果继续做，我会怎么推进

下一阶段如果继续往生产级走，我会优先补下面几块：

### 10.1 Poller 并发控制

- 多实例部署时，为 outbox 扫描增加分布式锁或 claim 机制
- 避免多个实例同时扫到并重复派发同一批 `NEW` 数据

### 10.2 outbox 更完整的生命周期

- 增加最大重试次数
- 增加 `FAILED / DEAD` 状态
- 落最后一次派发错误原因

### 10.3 超时任务补偿

- 扫描长时间停留在 `DISPATCHING / PROCESSING` 的任务
- 做状态修正、失败标记或人工排查入口

### 10.4 用户可感知的结果反馈

- 先做前端轮询状态
- 再考虑站内消息或更主动的通知方式

---

## 11. 最后的结论

如果只看表面，这次像是在做：

- 状态机优化
- MQ 异步解耦
- 幂等控制

但如果从底层看，这次真正的收获其实是：

**我开始更清楚地意识到，微服务里的很多复杂度，本质上都来自“流程不是一步完成的”。**

只要不是一步完成：

- 就会有中间态
- 就会有身份问题
- 就会有幂等问题
- 就会有“哪一步才算真相”的问题

这次从 `DISPATCHING` 到 `task / outbox / inbox` 的整个演进，其实就是在一步步承认这些事实，然后把它们显式建模出来。

这也让我更确定一点：

**系统设计不是为每个问题找最强答案，而是先把真正的矛盾暴露出来，再为当前阶段找到最合适、最可演进的答案。**
