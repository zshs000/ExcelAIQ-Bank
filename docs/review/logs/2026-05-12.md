### 1. ✅ 已解决 ~~`sendingTimeoutSeconds()` 方法缺少边界检查~~

**文件**: `eaqb-question-bank/eaqb-question-bank-biz/src/main/java/com/zhoushuo/eaqb/question/bank/biz/service/impl/QuestionOutboxDispatchScheduler.java`

```java
private int sendingTimeoutSeconds() {
    long timeoutMs = Math.max(sendingTimeoutMs, 1000L);
    long seconds = (timeoutMs + 999L) / 1000L;
    return seconds > Integer.MAX_VALUE ? Integer.MAX_VALUE : (int) seconds;
}
```

~~当前方法对 `sendingTimeoutMs` 做了最小 1000ms 的限制，但未处理配置为负数的情况。虽然 `Math.max(sendingTimeoutMs, 1000L)` 会处理负数，但建议在配置注入时添加 `@Value` 的默认值验证，或在方法中添加注释说明为何允许负数输入。~~

已解决：添加了 Javadoc 注释，说明非法或过小的配置按 1 秒处理。

### 2. ⚠️ 可不改 ~~`claimDispatchableEvent` 的 SQL 条件可能过于宽松~~

**文件**: `eaqb-question-bank/eaqb-question-bank-biz/src/main/resources/mapper/QuestionOutboxEventDOMapper.xml`

```xml
<update id="claimDispatchableEvent">
    update question_outbox_event
    set event_status = 'SENDING',
        next_retry_time = null,
        updated_time = now()
    where id = #{id,jdbcType=BIGINT}
      and (
        (
          event_status in ('NEW', 'RETRYABLE')
          and (next_retry_time is null or next_retry_time <= now())
        )
        or (
          event_status = 'SENDING'
          and updated_time is not null
          and updated_time <= date_sub(now(), interval #{sendingTimeoutSeconds,jdbcType=INTEGER} second)
        )
      )
</update>
```

~~当前条件允许抢占 `NEW`、`RETRYABLE` 和超时的 `SENDING` 状态事件。但在高并发场景下，如果多个实例同时扫描到同一个超时 `SENDING` 事件，它们都会尝试抢占，只有一个会成功。这是预期的行为，但需要确保下游处理是幂等的。~~

**用户确认可不改**：MySQL 单行 update 具有原子性，多实例同时抢占超时 SENDING 时，第一个实例更新成功后 `updated_time` 变为 `now()`，后续实例条件不满足，影响行数为 0。符合预期：同一轮只有一个实例抢到。剩余的"MQ 成功但本地未标记 SENT 后宕机"仍属于 at-least-once 设计边界。
