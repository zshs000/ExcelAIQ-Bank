## 已确认误判

以下项经用户确认为误判，审查时不再报：

| ID | 原审查项 | 误判原因 |
|----|---------|---------|
| W1 | `@ConditionalOnProperty(matchIfMissing=true)` | 生产环境消费者必须默认注册，静默不注册比本地报错危险得多 |
| W2 | `JsonUtils.OBJECT_MAPPER` 非 volatile | `init()` 仅在 Spring 启动时调用一次，之后只读，容器初始化有内存屏障 |
| W3 | Redis `increment` + `expireAt` 竞态 | `INCR` 是原子操作，全局只有一个请求拿到 `count==1`，不存在并发竞态 |
| W4 | `AuthController` 缺少 `@RequestMapping` | 网关 `StripPrefix=1` 设计，加了反而路径冲突 |
| W5 | `@SneakyThrows` 滥用 | SDK IO 异常走 `@SneakyThrows` 冒泡，全局异常处理器兜底，业务校验已用 `BizException` |
| W6 | `QuestionExcelListener` 异常丢失 | `chunkConsumer` 抛异常时 `buffer.clear()` 不执行，数据仍在 buffer，整条导入已失败 |
| W7 | `QuestionCallbackAppService` @Transactional 批量回滚 | 消费端每次只传入 `Map.of(单条)`，批量永远是 1，整批回滚前提不成立；且 inbox 唯一约束兜底 |
| W8 | `GlobalExceptionHandler` 各模块重复定义 | Gateway 用 WebFlux `ErrorWebExceptionHandler`，其余服务用 Servlet `@ControllerAdvice`，且各模块错误码前缀不同，提取反而引入模板复杂度 |
| W9 | `RedisTemplateConfig` 跨模块重复，建议收敛到 framework | 维护成本实际很低（3 模块各 20 行）；收敛会污染不需要 Redis 的模块的依赖树，且各模块未来可能需要不同配置 |
| W10 | `distributed-id-generator`、`eaqb-oss` 等内部服务的接口无鉴权或无归属校验 | 这些服务均未配置网关路由，是纯内部服务，外部用户无法直接访问，只能被其他微服务通过 Feign 调用。归属校验在上游业务服务中完成（如 `ExcelFileRecordSupport.loadOwnedFile()`）。审查时应先确认服务是否在网关路由中，再判断是否存在安全漏洞 |
