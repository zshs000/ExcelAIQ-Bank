# 代码审查指南

## 审查原则

1. 指出潜在问题,但区分"确定的缺陷"和"取决于上下文的取舍"。对于后者,说明"如果你认为当前场景下这不是问题,可以不改"。
2. AI 缺乏完整的业务上下文,最终判断权在用户。**上下文不足时，不要直接给出最严厉的审查意见（如 P0/高优先级）**，应标注"取决于上下文"或主动询问用户确认。
3. 遇到 `已知上下文.md` 中记录的项,引用上下文并询问"是否仍维持之前的判断",不重复当作新问题报。
4. 已解决的项用删除线标记，误判的项移到“误判”区域并说明原因。
5. **编号规则**：新增 K 或 W 条目前，必须先读取 `已知上下文.md` 确认当前最大编号，避免重复。你不是唯一的工作线程，其他会话可能已经添加了新条目。
6. **安全类问题审查前置**：审查鉴权、权限、归属校验等问题时，应先确认目标服务是否在网关路由中（即是否对外暴露），避免将内部服务的设计模式误判为安全漏洞。

## 审查流程

1. 读 `docs/review/已知上下文.md`,了解已知项与历史决策。
2. 审查代码,收集发现。
3. 对照已知上下文,匹配项引用上下文,不重复报。
4. 按格式输出审查意见。
5. 审查日志存放到 `docs/review/logs/{日期}.md`。

## 审查格式

按严重程度分三级,每条包含文件位置、代码片段、说明:

```
## 🔴 高优先级

### 1. 问题标题

**文件**: `模块路径/.../文件名.java`

```java
// 问题代码片段
```

说明文字。
```

### 状态标记

| 标记 | 含义 |
|------|------|
| 无标记 | 待处理 |
| `✅ 已解决` | 用户已修复 |
| `⚠️ 可不改` | 上下文取舍,非必须修复 |
| `❌ 误判` | 审查错误,非问题 |

已解决和误判的项保留在文档中,用删除线标记,附上原因。误判项统一放在"误判"区域。

### 汇总表

文档末尾附汇总:

```
| 状态 | 数量 |
|------|------|
| 🔴🟡🟢 待处理 | N |
| ✅ 已解决 | N |
| ⚠️❌ 降级/误判 | N |
```

## 已知上下文的维护

审查过程中,当出现以下情况时,AI 应主动询问用户是否写入 `已知上下文.md`:

1. **用户指出误判**:用户解释了某条审查意见不是问题(例如"这个是有意为之""这个不影响"),AI 应询问"是否要将这条误判原因写入已知上下文,避免后续审查重复报出?"
2. **用户解释权衡**:用户说明了当前设计的取舍理由(例如"现在单实例部署,分布式锁暂不需要"),AI 应询问"是否要将这个权衡记录到已知上下文中?"
3. **用户确认可不改**:用户同意某条建议但决定暂不修复,有明确的上下文原因时,AI 应主动建议记录。

写入时同步更新 `已知上下文.md` 对应表格,并在审查日志中引用新增的上下文编号。

### 删除或修改条目时的编号维护

当删除或移动 `已知上下文.md` 中的条目时,必须同步检查并重新编号后续条目,保持编号连续。例如删除 K4 后,原 K5 应改为 K4、K6 改为 K5,依此类推。

## 已知上下文的引用方式

当审查发现匹配 `已知上下文.md` 中的项时,在该条末尾追加:

> 已知上下文 K{n}:{简要引用上下文内容}。是否仍维持之前的判断?

示例:

> **Outbox 调度器缺少分布式锁**
> 已知上下文 K1:当前单实例部署,暂不处理。是否仍维持之前的判断?
