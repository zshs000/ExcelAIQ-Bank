# 代码审查指南

## 审查原则

1. 指出潜在问题，但区分"确定的缺陷"和"取决于上下文的取舍"。对于后者，说明"如果你认为当前场景下这不是问题，可以不改"。
2. AI 缺乏完整的业务上下文，最终判断权在用户。
3. 遇到 `已知上下文.md` 中记录的项，引用上下文并询问"是否仍维持之前的判断"，不重复当作新问题报。
4. 已解决的项用删除线标记，误判的项移到"误判"区域并说明原因。

## 审查流程

1. 读 `docs/review/已知上下文.md`，了解已知项与历史决策。
2. 审查代码，收集发现。
3. 对照已知上下文，匹配项引用上下文，不重复报。
4. 按格式输出审查意见。
5. 审查日志存放到 `docs/review/logs/{日期}.md`。

## 审查格式

按严重程度分三级，每条包含文件位置、代码片段、说明：

```
## 🔴 高优先级

### 1. 问题标题

**文件**: `模块路径/.../文件名.java`

```java
// 问题代码片段
```

说明文字。
```

### 状态标记

| 标记 | 含义 |
|------|------|
| 无标记 | 待处理 |
| `✅ 已解决` | 用户已修复 |
| `⚠️ 可不改` | 上下文取舍，非必须修复 |
| `❌ 误判` | 审查错误，非问题 |

已解决和误判的项保留在文档中，用删除线标记，附上原因。误判项统一放在"误判"区域。

### 汇总表

文档末尾附汇总：

```
| 状态 | 数量 |
|------|------|
| 🔴🟡🟢 待处理 | N |
| ✅ 已解决 | N |
| ⚠️❌ 降级/误判 | N |
```

## 已知上下文的引用方式

当审查发现匹配 `已知上下文.md` 中的项时，在该条末尾追加：

> 已知上下文 K{n}：{简要引用上下文内容}。是否仍维持之前的判断？

示例：

> **Outbox 调度器缺少分布式锁**
> 已知上下文 K1：当前单实例部署，暂不处理。是否仍维持之前的判断？
