# Excel解析与题目处理链路梳理

## 1. 先用一句话说清楚这条链路

这条链路本质上只做两件事：

1. 把 Excel 里的题目安全地导入系统
2. 把导入后的题目送去 AI 处理，再回到人工审核

如果你觉得它乱，通常不是因为业务本身复杂，而是因为：

- 文件上传和文件解析被拆成了两段
- 题目录入和 AI 处理又被拆成了两段
- 中间穿插了 `oss`、`id-generator`、`question-bank`、`MQ`

所以理解它的关键不是记模块，而是记住它的 4 个阶段：

1. 上传并校验 Excel
2. 解析 Excel 并批量导入题目
3. 发送题目到 AI 异步处理
4. AI 回包后更新题目状态，进入人工审核

## 2. 用户视角的完整流程

### 阶段 A：上传 Excel

用户上传一个 Excel 文件。

系统会做：

- 文件大小校验
- 文件扩展名校验
- 文件魔术头校验
- Excel 模板内容校验

如果校验失败：

- 返回失败摘要
- 记录一条预上传记录
- 支持按 `preUploadId` 查询详细错误信息

如果校验成功：

- 把文件上传到对象存储
- 保存文件记录
- 返回 `fileId`

### 阶段 B：按文件 ID 解析并导入题目

用户拿着 `fileId` 再触发一次解析。

系统会做：

- 根据 `fileId` 查文件记录
- 校验当前用户是不是文件拥有者
- 通过 `oss` 服务获取文件访问链接
- 下载 Excel 文件流
- 使用 EasyExcel 解析题目数据
- 转成题库服务需要的 DTO
- 调用 `question-bank` 批量导入

批量导入成功后：

- 每道题生成一个全局唯一 ID
- 题目写入数据库
- 初始状态统一为 `WAITING`

当前返回语义：

- 文件不存在 / 无权限：外层直接返回失败
- 文件下载或解析异常：外层返回失败
- 题库导入失败：外层调用成功，但 `ExcelProcessVO.processStatus = FAILED`

也就是说，当前系统把“是否成功进入解析流程”和“题库导入是否成功”区分成了两层语义。

当前文件状态流转：

- 上传成功：`UPLOADED`
- 开始解析：`PARSING`
- 解析并导入成功：`PARSED`
- 解析或导入失败：`FAILED`

### 阶段 C：发送题目到 AI 处理

题目录入成功以后，题目只是进入系统了，还没有进入 AI 处理。

此时用户可以手动触发“发送到队列”。

系统会做：

- 校验题目是否属于当前用户
- 校验题目当前状态是否允许发送
- 根据 `mode` 决定哪些题目能发
  - `GENERATE`：只发送没有答案的题目
  - `VALIDATE`：只发送已有答案的题目
- 把题目状态从 `WAITING` 改成 `PROCESSING`
- 发送 RocketMQ 消息给 AI

如果本地开发环境没开 MQ，但开了 `mock-enabled`：

- 不发 MQ
- 本地直接模拟 AI 成功
- 状态从 `PROCESSING` 推到 `REVIEW_PENDING`

### 阶段 D：AI 回包并进入审核

AI 处理完成后，会往结果 Topic 回包。

系统收到回包后：

- 成功：把题目从 `PROCESSING` 推进到 `REVIEW_PENDING`
- 失败：把题目从 `PROCESSING` 推进到 `PROCESS_FAILED`

其中成功场景下：

- `GENERATE`：默认把 AI 生成的答案写入 `answer`
- `VALIDATE`：默认不覆盖原答案，只推进状态，等待人工审核

### 阶段 E：人工审核

AI 处理成功后，题目不会直接变成最终完成，而是进入 `REVIEW_PENDING`。

此时用户可以：

- `APPROVE`：通过，状态进入 `COMPLETED`
- `REJECT`：驳回，状态回到 `WAITING`

这一步的作用是：

- AI 只做辅助，不直接决定最终结果
- 人工仍然保留最后确认权

## 3. 服务调用视角的完整链路

从服务角度看，这条链路可以画成：

`gateway -> excel-parser -> oss -> excel-parser -> question-bank -> rocketmq -> AI -> rocketmq -> question-bank`

更细一点：

1. 前端通过网关调用 `excel-parser/upload`
2. `excel-parser` 校验文件，调用 `oss` 上传文件
3. `excel-parser` 保存文件记录
4. 前端再调用 `excel-parser/parse-by-id`
5. `excel-parser` 通过 `oss` 获取文件访问链接并下载文件
6. `excel-parser` 解析 Excel
7. `excel-parser` 调 `question-bank` 批量导入
8. 用户调用 `question-bank/send-to-queue`
9. `question-bank` 发消息到 RocketMQ
10. AI 服务消费消息并回包
11. `question-bank` 消费回包并更新题目状态
12. 用户调用审核接口完成最终流转

## 4. 这条链路为什么让人容易混乱

主要有 4 个原因。

### 4.1 文件上传和文件解析不是同一个动作

很多人会下意识以为：

- 上传 Excel = 立刻解析并入库

但这个项目不是这样。

它是：

- 第一步只负责“校验 + 上传 + 存记录”
- 第二步才负责“按 fileId 解析并导入”

这会让第一次看代码的人产生割裂感。

### 4.2 Excel 服务既处理文件，又在编排题目录入

`excel-parser` 并不只是一个“读 Excel”的工具服务，它还在做流程编排：

- 调对象存储
- 调题库服务
- 调 ID 服务

所以它不是纯解析器，而是“导入流程编排者”。

### 4.3 题目录入和 AI 处理是两段业务

题目入库后状态只是 `WAITING`，并不会自动送去 AI。

也就是说：

- “导入成功” 不等于 “AI 已处理”

这是第二个容易混的点。

### 4.4 题目状态机没有和导入链路天然写在一起

导入逻辑主要在 `excel-parser`

状态流转逻辑主要在 `question-bank`

所以你如果按“代码目录”理解，很容易被拆散。

## 5. 正确的心智模型

最简单的记法是：

### 第一段：导入

`Excel -> 校验 -> 上传 -> 下载 -> 解析 -> 批量入库 -> WAITING`

### 第二段：处理

`WAITING -> 发送AI -> PROCESSING -> 回包 -> REVIEW_PENDING/PROCESS_FAILED`

### 第三段：审核

`REVIEW_PENDING -> APPROVE/REJECT -> COMPLETED/WAITING`

你只要记住这三段，就不会乱。

## 6. 现在这条链路里真正比较合理的部分

- Excel 上传前做了多层校验
- 文件和题目是分阶段处理，不会一上来把所有逻辑塞在一个接口里
- 题目状态机是清晰的
- AI 处理用 MQ 解耦是合理的
- 人工审核保留在最终环节，也合理

## 7. 现在这条链路里真正不够好的部分

- `excel-parser` 承担的编排职责偏重
- 上传后再从对象存储下载回来解析，链路偏长
- 整个流程对“先上传、再解析”的理解成本较高
- 微服务拆分相对当前业务复杂度来说偏重

## 8. 面试时怎么讲这条链路

推荐讲法：

> 这条链路我后来重新梳理过，本质上是“导入”和“处理”两段业务。  
> 第一段是 Excel 导入：先做文件校验和上传，再按文件 ID 下载解析并批量导入题目，题目初始状态为 `WAITING`。  
> 第二段是题目处理：用户手动把题目发送给 AI，状态进入 `PROCESSING`；AI 回包后，系统根据结果推进到 `REVIEW_PENDING` 或 `PROCESS_FAILED`，最后再由人工审核决定是否完成。  
> 我后面最大的反思是，这条链路的业务并不复杂，但服务拆分和流程编排一开始做得偏重，所以我后续开始把它收敛成更容易理解和维护的形式。

## 9. 一句话结论

这条链路不是“特别复杂”，而是“被拆得比较散”。

只要你把它重新收敛成：

- 导入
- AI 处理
- 人工审核

这三个阶段，你就能讲清楚了。
